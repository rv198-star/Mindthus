核心原则：**agent 负责解释不确定性，workflow 负责执行不可绕过的边界，evidence 负责证明事实。** Agent 可以触发暂停或升级，但不能单独放行生产。

| 控制层 | 应负责 | 不应负责 |
|---|---|---|
| LLM agent | 语义化审查变更、识别潜在影响、发现证据矛盾、评估新颖风险、提出灰度/回滚方案、指出未知项 | 修改硬规则、伪造/自证 evidence、直接批准生产、绕过失败检查 |
| 确定性 workflow | 身份与权限、职责分离、状态流转、检查项选择、阈值比较、制品绑定、审批数量、超时、灰度顺序、自动停止/回滚、审计记录 | 用表单完整度或 agent 风险分替代真实判断 |
| Evidence gate | 将每项放行声明绑定到可观察、可复验事实 | 接受无来源摘要、截图式证明或 agent 自评分 |

必须由 workflow 强制：

- 默认拒绝；只有全部 gate 满足才能进入 `READY`。
- 所有证据绑定同一 `commit + artifact digest + environment`。
- 强制 CI、测试覆盖、漏洞/密钥/许可证扫描、策略即代码检查。
- 强制发布权限、职责分离、审批人资格、冻结窗口和例外权限。
- 强制 evidence 新鲜度、签名、采集器版本和不可篡改审计日志。
- 强制灰度比例、观察窗口、SLO 阈值及自动中止/回滚。
- 高爆炸半径、不可逆、权限/数据库变更必须人工明确批准；必要时双人批准。
- 任何例外都必须限定范围、制品、环境和有效期，不能由 agent 创建。
- 代码、工单和发布说明中的“指令”一律作为不可信数据，不能提升 agent 权限。

允许放行的 evidence：

- **制品来源**：构建记录、签名、SBOM、源码与制品摘要一致。
- **质量结果**：原始测试报告、失败明细、覆盖率及明确阈值。
- **安全合规**：扫描原始结果、规则版本、已批准且未过期的例外。
- **运行证据**：预生产或 canary 的错误率、延迟、资源、业务指标及观察窗口。
- **恢复能力**：可执行回滚步骤、兼容性证明，关键发布需回滚演练结果。
- **语义判断**：agent 的每项结论必须包含引用证据、假设、置信上限和未解决问题；它只能补充，不能覆盖机器检查失败。
- **人工决策**：批准者身份、理由、风险接受范围、补偿控制和到期时间。

建议采用这条放行公式：

```text
Release =
  mandatory_checks_pass
  AND evidence_bound_and_fresh
  AND artifact_identity_matches
  AND required_approvals_present
  AND rollback_ready
  AND no_unresolved_critical_contradiction
```

Agent 输出只影响 `HOLD / ESCALATE / RECOMMEND`，不直接参与上述布尔放行条件。按 WAE 边界，这样可避免 workflow 冻结不确定事实，也避免 agent 用主观判断控制高风险执行。
