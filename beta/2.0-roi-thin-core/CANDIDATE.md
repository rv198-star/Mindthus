# Mindthus 2.x ROI Thin Core Candidate

Status: implementation candidate; not a release or publication artifact.

## Claim

Keep the complete 1.4.6 owner topology, but replace the high-frequency
`using-mindthus` body with a small shared judgment floor. Accept loss only when the
missed activation would not change the decision or execution. The candidate succeeds
only if that bounded loss buys a material reduction in actual loading cost.

## Single Allowed Product Delta

- Keep all seven owner Skills and their descriptions, bodies, resources, and native
  Codex discoverability byte-identical to 1.4.6.
- Replace only `skills/using-mindthus/SKILL.md` at candidate assembly time.
- Keep the Stable Codex `defaultPrompt` unchanged for this first causal candidate.
- Add no Hook, injected project instructions, model-name branch, second entry, owner
  index, hidden atlas, or semantic routing script.
- Keep the 1.4.6 `using-mindthus` resources in the package as compatibility/reference
  material, but do not reference or preload them on the normal Thin Core path.

This isolates the question: does deleting methodology detail from the frequently loaded
entry improve ROI without losing decision-changing behavior?

## Loss Budget

Acceptable loss is classified by effect, not by a raw recall percentage.

Allowed:

- missing a method-aware refinement that only improves terminology, explanation depth,
  formatting, or analytical completeness;
- answering directly when loading a formal method would not change the action.

Rejected:

- missing an explicit or clearly discovered owner;
- changing the correct strategy, evidence requirement, risk treatment, authority,
  next action, or stopping condition;
- adding a user-visible routing turn on ordinary or near-negative work;
- claiming savings from source size when host token evidence does not distinguish the
  candidate.

One decision-changing regression blocks continuation; harmless enrichment misses do not.

## ROI Gates

Static qualification requires:

- Thin Core at most 2,300 bytes and at most 35% of the 1.4.6 entry size;
- all seven owner trees byte-identical to the 1.4.6 baseline;
- no methodology names, owner table, conditional-resource preload, Hook, model branch,
  or extra runtime carrier in the active candidate delta.

Live comparison requires matched Codex model, effort, prompts, isolation, and plugin
state. Continue only when:

- explicit owner tasks and decision-changing passive cases retain the correct action;
- ordinary and near-negative cases add neither unrelated owner loads nor interaction
  rounds;
- median actual Mindthus loaded bytes fall by at least 50%; and
- host-reported median uncached input tokens fall by at least 10%, or the result is
  honestly recorded as cost-unproven.

Latency is supporting evidence because host and network noise can dominate a small run.
It may strengthen a decision, but source bytes cannot substitute for token or lifecycle
evidence.

## Stop Conditions

Stop this candidate after one decision-changing regression, missing trustworthy
activation evidence, or an indistinguishable cost result. Do not add a Hook, owner index,
fallback router, or method summary to rescue it. A failed Thin Core does not authorize a
new routing architecture.
