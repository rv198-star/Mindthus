# Mindthus H2.1 终局决策报告

## 结论

本轮结论是：`STOP_2X_EXPLORATION`。

更直白地说：这次“保留 H2 单入口拓扑，只收紧 MPG→SELA companion gate”
的最后一次受限修正，解决了旧反例的一半，但没有证明完整因果链。因此当前 2.x
路由探索不能继续进入正式对照，更不能据此发布。

这不是说所有未来 2.x 架构在逻辑上都不可能，而是说我们预先允许的 H1、H2 和
H2.1 路线已经用完，继续需要再改 owner 选择权、反向握手或路由结构，实质上已经是
新的架构假设。按照任务边界，此处应停止，而不是把失败继续包装成下一轮修补。

## 已经证明了什么

1. H2.1 静态候选是可复现的：只有一个可发现入口、七个按需资源 owner，没有 Hook、
   AGENTS 注入、defaultPrompt、第二入口、模型名分流或语义路由脚本。
2. Stable 正式 owner 源码没有改动；构建时只替换 MPG 的两处同义 companion 声明，
   总计减少 102 bytes，其余 owner 反归一化后全部命中 1.4.6 冻结锁。
3. 完整仓库 646 个测试通过，最终构建诊断 30/30 通过，三路静态审计通过。
4. C1/Q1 正确地只读薄入口，在缺少答案关键上下文时没有贸然给方向。
5. C1/Q2 正确地只读 index 和 MPG，没有再误读 SELA。也就是说，本次收窄确实消除了
   促成本轮修正的那个旧 false-positive。

## 为什么仍然失败

C2 专门验证真正需要 SELA+MPG 的正例。预注册要求是：

```text
入口 -> index -> MPG -> SELA
```

真实生命周期却是：

```text
入口 -> index -> SELA -> MPG
```

差别不是形式问题。真实执行先由 index 命中 SELA，再由未修改的 SELA 反向握手拉起
MPG；本轮修改的 MPG companion gate 根本没有承担正向触发。因此我们只能证明
“负例不再误唤醒”，不能证明“正例仍会由修正后的 MPG gate 正确唤醒 SELA”。

C2 最终业务答案其实不错：它保留三年共享平台方向，同时暂停德国切换，并给出恢复和
替换条件。但本 Mission 一开始就规定，漂亮答案不能抵消路由生命周期失败；否则测试
将无法回答“修改是否真的有效”这个因果问题。

## 为什么不继续跑 C3 和 Stable

C3 只能再证明一个 near-negative 不读 SELA，无法补上已经缺失的正向因果证据。
Q3-Q7 和 Stable/Judge 都只允许合格候选进入；继续跑只会花钱测试一个已经失去资格的
候选。

若要强制 MPG 先拥有 C2，再按 companion gate 读取 SELA，至少要继续修改 index 的
owner 优先关系、SELA 的反向握手或 owner 仲裁规则。这已超出“一次 MPG gate 减法修正”
授权，也会重新打开我们刻意禁止的 guardrail 迭代。

## 最终定位

- 1.4.6 继续作为功能完整基线。
- H2.1 分支和证据保留为“静态成立、负向修正有效、正向因果失败”的拒绝样本。
- 不运行 C3、Q3-Q7、Stable、Judge 或噪声复核。
- 不发布、不合并、不打 tag、不建 Beta.3、不改长期 CODEX_HOME、不准备 release。
- 当前 Mission 到此合法完成；若未来重开，必须作为新的产品/架构探索，而不是声称本轮
  还有一个小补丁即可通过。
