可以从“权限、流程、话术、监控”四层治理，不要只靠提示词约束。

1. **收回直接承诺权限**
   - Agent 不应能直接发送“已退款 / 一定退款 / 会给你退”等承诺性话术。
   - 把退款动作拆成两类：
     - `refund_request_create`：agent 可调用，只是创建退款申请；
     - `refund_approve_or_execute`：只能由人工或审批服务调用。

2. **把退款政策做成硬规则**
   不要让模型自己判断“该不该退”。应由后端策略服务判断，例如：
   - 订单状态；
   - 付款时间；
   - 是否已发货 / 已使用；
   - 金额阈值；
   - 用户历史退款次数；
   - 是否命中风控。

   Agent 只能拿到结果：
   - `eligible_auto_refund`
   - `needs_human_review`
   - `not_eligible`

3. **强制人工审批节点**
   对高风险场景直接锁死：
   - 超过某金额；
   - 争议订单；
   - 老客高频退款；
   - 政策边界模糊；
   - 客户情绪激烈但事实不清。

   Agent 只能回复：“我已为你提交退款申请，预计由专员在 X 时间内审核。”

4. **限制可用话术**
   给 agent 明确允许和禁止表达。

   允许：
   - “我可以帮你提交退款申请。”
   - “该申请需要人工审核。”
   - “审核通过后会按原支付方式退回。”
   - “目前我还不能确认退款结果。”

   禁止：
   - “一定给你退。”
   - “已经安排退款。”
   - “我保证可以退款。”
   - “不用审核。”

5. **上线输出拦截**
   在 agent 回复发给客户前加一层检测：
   - 如果包含退款承诺语义；
   - 且当前没有审批通过记录；
   - 则阻断发送，改写为“提交申请 / 等待审核”版本。

6. **保留审计日志**
   每次涉及退款都记录：
   - 用户问题；
   - agent 判断；
   - 调用过的工具；
   - 是否触发人工审批；
   - 最终给客户的回复；
   - 是否发生违规承诺。

   这样才能定位是提示词问题、工具权限问题，还是审批状态同步问题。

7. **用测试集回归**
   建一批专门的退款对话测试：
   - 客户强烈要求马上退款；
   - 客户威胁投诉；
   - 边界政策案例；
   - 高金额订单；
   - 已使用服务；
   - 人工审批未完成；
   - 审批被拒绝。

   每次改 prompt、模型或工具权限后都跑一遍。

核心原则是：**模型可以解释政策和提交申请，但不能单独决定或承诺退款。退款承诺必须来自确定的审批状态，而不是来自模型生成。**
