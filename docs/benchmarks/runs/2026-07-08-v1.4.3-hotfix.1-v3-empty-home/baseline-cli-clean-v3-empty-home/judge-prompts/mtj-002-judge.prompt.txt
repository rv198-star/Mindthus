# 事故复盘报告

## 一、事故概述

本次故障主要表现为业务请求超时、部分接口响应延迟升高、服务间调用失败率上升。结合故障期间的监控数据、业务日志和链路表现，初步判断本次事故的主要触发因素为网络抖动，导致服务间通信不稳定，进而引发上层业务异常。

## 二、影响范围

事故期间，部分用户访问出现响应慢、请求失败或重试后成功的情况。受影响范围主要集中在依赖跨节点、跨服务通信的业务链路中，对单机本地处理逻辑影响较小。

## 三、事故现象

故障发生时，系统出现以下特征：

1. 多个服务同时出现接口超时，但服务进程本身未发生大面积崩溃。
2. 应用 CPU、内存、磁盘 IO 等资源未出现持续性瓶颈。
3. 服务间调用耗时明显增加，部分 RPC / HTTP 请求出现连接超时、读超时或重试。
4. 故障呈现间歇性、波动性特征，符合网络抖动造成的不稳定表现。
5. 业务恢复后，服务本身无需大规模重启即可逐步恢复正常。

## 四、原因分析

本次事故的主要原因判断为网络链路在故障时间窗口内出现抖动，导致服务间通信质量下降。网络抖动可能造成短时间丢包、延迟升高、连接建立失败或已有连接异常，从而影响上层服务调用。

由于当前业务系统存在较多服务间同步调用，一旦网络质量下降，调用链路上的超时和重试会被放大，进一步造成请求堆积、线程占用、连接池资源紧张等连锁反应。

本次故障暴露出以下问题：

1. 对网络抖动场景的监控颗粒度不足。
2. 服务调用链路对短时网络异常的容错能力不够。
3. 部分接口超时和重试策略不够精细，存在故障放大的风险。
4. 缺少针对网络异常的快速定位和隔离手段。

## 五、处置过程

故障发生后，运维和研发人员对业务状态、主机资源、应用日志、调用链路和网络连通性进行了排查。排查过程中发现，业务异常集中体现为服务间通信超时和延迟升高，而应用自身资源未出现明显瓶颈。

在确认故障与网络质量波动高度相关后，团队重点关注网络链路、节点连通性、丢包率、延迟变化及相关基础设施状态。随着网络状态恢复，业务请求成功率逐步回升，系统整体恢复正常。

## 六、改进措施

1. 增强网络监控  
   补充节点间延迟、丢包率、TCP 重传、连接失败率等指标监控，并建立告警阈值。

2. 优化超时和重试策略  
   对关键服务调用设置合理超时时间，避免过长等待；重试策略增加退避机制，防止故障期间请求被放大。

3. 提升服务容错能力  
   对核心链路增加熔断、限流、降级能力，避免局部网络异常影响整体业务。

4. 完善链路追踪  
   强化调用链路中的网络耗时分析能力，便于快速区分应用问题、资源问题和网络问题。

5. 建立网络故障演练机制  
   定期模拟延迟升高、丢包、连接失败等场景，验证系统在网络抖动下的稳定性。

## 七、结论

本次事故主要由网络抖动引发，网络质量下降导致服务间调用超时和失败，并在复杂调用链路中被进一步放大。后续需要围绕网络可观测性、调用容错、超时重试策略和故障演练进行改进，降低类似问题再次发生时对业务的影响。
