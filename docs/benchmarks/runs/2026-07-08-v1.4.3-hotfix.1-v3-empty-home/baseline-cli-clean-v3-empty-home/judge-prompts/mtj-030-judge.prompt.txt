# 项目风险评估

## 一、技术风险

### 风险 1：核心功能实现难度高，导致开发延期
**风险描述：**  
项目中若存在复杂业务规则、第三方系统对接、性能要求或历史系统兼容问题，可能导致核心功能开发时间超出预期。

**触发信号：**
- 核心模块连续 2 个工作日无实质进展；
- 接口联调超过计划时间仍无法稳定通过；
- 技术方案频繁变更；
- 关键功能需要返工。

**影响：**
- 开发排期延后；
- 测试时间被压缩；
- 上线质量下降。

**应对措施：**
- 在项目早期完成技术方案评审；
- 对高风险模块先做技术验证或 Demo；
- 将复杂功能拆分为“必须上线”和“可延后优化”两类；
- 每周跟踪核心模块完成率，而不是只看整体进度。

**负责人：** 技术负责人  
**检查频率：** 每周至少一次，联调阶段每两天一次。

---

## 二、市场风险

### 风险 2：需求与市场变化不匹配，导致上线后价值不足
**风险描述：**  
如果项目周期内用户需求、竞品策略或业务优先级发生变化，可能导致已开发功能无法满足实际市场需要。

**触发信号：**
- 业务方频繁提出需求调整；
- 竞品上线相似功能；
- 用户反馈与当前方案存在明显偏差；
- 原定业务目标发生变化。

**影响：**
- 产品上线后使用率低；
- 已投入开发资源浪费；
- 项目需要二次改版。

**应对措施：**
- 明确本期项目的核心目标，例如提升转化率、降低人工处理成本、提高留存等；
- 每两周与业务方确认一次需求优先级；
- 对非核心需求设置冻结时间，避免临近上线频繁变更；
- 保留小范围灰度发布或快速调整空间。

**负责人：** 产品负责人 / 业务负责人  
**检查频率：** 每两周一次，重大市场变化时立即评估。

---

## 三、人员风险

### 风险 3：关键人员负荷过高，影响进度和质量
**风险描述：**  
如果核心开发、测试、产品或业务确认人员同时承担多个任务，可能出现响应慢、决策延迟、交付质量下降等问题。

**触发信号：**
- 关键问题超过 1 个工作日无人响应；
- 需求确认、接口确认、测试反馈反复延期；
- 同一人员承担多个关键路径任务；
- 缺陷修复速度明显下降。

**影响：**
- 任务阻塞；
- 沟通成本增加；
- 项目延期或上线质量不稳定。

**应对措施：**
- 明确关键岗位备份人；
- 将关键任务拆分到具体人和具体日期；
- 每日识别阻塞事项，不等到周会才处理；
- 对非关键任务进行降级、延期或转交。

**负责人：** 项目经理  
**检查频率：** 每日站会或每日进度同步。

---

## 四、交付风险

### 风险 4：测试和验收时间不足，导致带病上线
**风险描述：**  
项目周期紧张时，开发延期往往会挤压测试、修复和验收时间，增加上线故障风险。

**触发信号：**
- 开发完成时间晚于计划超过 20%；
- 测试阶段仍有大量需求变更；
- 上线前仍存在高优先级缺陷；
- 验收人员未按时完成确认。

**影响：**
- 上线后出现严重缺陷；
- 用户体验受损；
- 需要紧急回滚或补丁发布。

**应对措施：**
- 设定上线准入标准，例如无 P0/P1 缺陷、核心流程测试通过、回滚方案确认；
- 提前准备测试用例和验收清单；
- 对低优先级功能允许延期上线；
- 制定上线回滚方案和应急联系人名单。

**负责人：** 项目经理 / 测试负责人 / 运维负责人  
**检查频率：** 测试阶段每日检查，上线前完成专项评审。

# 上线前必须确认

1. 核心功能是否全部完成并通过测试；
2. 是否仍存在高优先级缺陷；
3. 业务方是否完成验收确认；
4. 是否有明确的回滚方案；
5. 上线后是否安排监控和值班人员。

这版文档的重点不是“写得完整”，而是能用于项目周会、风险跟踪和上线决策。
