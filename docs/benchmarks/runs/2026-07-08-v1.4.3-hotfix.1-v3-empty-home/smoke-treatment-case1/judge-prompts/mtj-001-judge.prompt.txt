**建索引方案**

1. 先锁定 P99 相关 SQL
   - 取最近 1-3 小时慢查询日志、APM trace、数据库 `pg_stat_statements` / MySQL slow log。
   - 只处理：

     - P95/P99 高
     - 扫描行数远大于返回行数
     - `EXPLAIN` 出现全表扫、回表过多、排序临时表、filesort、hash join 异常的 SQL

2. 索引优先级
   - 第一优先级：高频读接口的 `WHERE + ORDER BY + LIMIT`
   - 第二优先级：核心 join 条件
   - 第三优先级：唯一性校验、状态流转查询
   - 暂不优先：低频报表、离线查询、一次性后台任务

3. 索引设计规则
   - 组合索引顺序：
     - 等值过滤字段在前
     - 范围字段放后
     - 排序字段尽量接在过滤字段后
   - 常见模式：

```sql
-- WHERE tenant_id = ? AND status = ? ORDER BY created_at DESC LIMIT ?
CREATE INDEX idx_table_tenant_status_created
ON table_name (tenant_id, status, created_at DESC);

-- WHERE user_id = ? AND deleted_at IS NULL ORDER BY id DESC
CREATE INDEX idx_table_user_deleted_id
ON table_name (user_id, deleted_at, id DESC);

-- JOIN: a.order_id = b.id
CREATE INDEX idx_child_order_id
ON child_table (order_id);

-- WHERE tenant_id = ? AND type = ? AND created_at BETWEEN ? AND ?
CREATE INDEX idx_table_tenant_type_created
ON table_name (tenant_id, type, created_at);
```

4. 避免乱建
   - 不给低选择性单字段随便建索引，例如单独的 `status`、`is_deleted`。
   - 不重复建左前缀已覆盖的索引：
     - 已有 `(tenant_id, status, created_at)` 时，一般不再建 `(tenant_id, status)`。
   - 控制单表索引数量，优先合并能覆盖多个核心查询的组合索引。
   - 写入高峰表要特别谨慎，索引会增加写入成本。

5. 上线方式
   - PostgreSQL：

```sql
CREATE INDEX CONCURRENTLY idx_xxx ON table_name (...);
```

   - MySQL/InnoDB：

```sql
ALTER TABLE table_name
ADD INDEX idx_xxx (...),
ALGORITHM=INPLACE,
LOCK=NONE;
```

6. 验证标准
   - 建索引前后对比同一 SQL 的：
     - `EXPLAIN ANALYZE`
     - 扫描行数
     - 临时表 / filesort
     - 平均延迟、P95、P99
     - 数据库 CPU、IO、buffer 命中率
   - 目标不是“命中索引”，而是接口 P99 回到 SLO 内。

7. 回滚策略
   - 每次只上一小批高置信索引。
   - 观察 30-60 分钟。
   - 如果写入延迟、锁等待、CPU 或磁盘 IO 明显恶化，删除新增索引：

```sql

```
