# 事故复盘：网络抖动导致服务异常

## 一、事故概述

在本次故障中，业务系统出现访问超时、请求失败率上升、部分服务间调用异常等现象。根据故障期间的监控表现、链路调用情况及运维侧排查结果，故障主要集中在网络链路稳定性下降，表现为短时间网络抖动、丢包或延迟升高，进而影响服务间通信质量。

本次事故未发现明显的应用发布、配置变更或容量突增直接触发因素，初步判断网络异常是主要诱因。

## 二、影响范围

事故期间受影响范围包括：

- 部分用户请求响应变慢或失败；
- 服务间 RPC / HTTP 调用出现超时；
- 依赖网络通信的组件出现间歇性异常；
- 个别业务流程因下游调用失败出现重试或中断。

## 三、故障表现

故障期间主要现象如下：

- 服务错误率明显升高；
- 接口 P95 / P99 延迟上升；
- 部分节点到依赖服务的连接出现间歇性超时；
- 应用日志中出现大量 `timeout`、`connection reset`、`connection refused` 或类似网络异常；
- 重启应用进程后问题未完全消失，说明问题不集中在单个应用实例。

这些现象符合网络链路抖动导致上层服务不稳定的典型特征。

## 四、原因分析

本次故障的直接原因是故障窗口内网络链路稳定性下降，导致服务之间通信质量劣化。由于微服务之间存在较多同步调用，网络延迟和丢包被放大为接口超时、重试增加和错误率上升。

从现象看，故障并非单点应用逻辑异常，而是多个服务在相近时间段出现跨服务调用异常，具备明显的网络层问题特征。

可能原因包括：

- 机房、交换机、云网络或专线链路短时抖动；
- 节点到网关、负载均衡或依赖服务之间链路质量下降；
- 网络设备转发表、连接跟踪或安全策略短时异常；
- 网络拥塞导致延迟升高和请求排队。

最终根因仍需结合网络设备日志、云厂商事件记录、丢包率、TCP 重传率、链路延迟等数据进一步确认。

## 五、处置过程

故障发生后，运维侧进行了以下处理：

1. 通过监控确认故障集中在接口超时和服务间调用异常；
2. 排查近期发布、配置变更及资源使用情况，未发现明显异常；
3. 对受影响服务进行节点、实例和链路维度排查；
4. 结合日志判断异常主要表现为网络连接超时；
5. 通过切换节点、摘除异常实例、调整流量或等待网络恢复后，业务逐步恢复。

## 六、暴露问题

本次事故暴露出以下问题：

- 网络层监控粒度不足，对丢包、TCP 重传、链路延迟缺少直接告警；
- 应用侧对网络抖动的容错能力不足，超时、重试、熔断策略仍需优化；
- 故障初期定位依赖人工经验，缺少从应用异常快速关联到网络指标的诊断链路；
- 多服务同步调用较多，网络异常容易被放大为业务级故障。

## 七、改进措施

后续重点从网络监控、应用容错和故障定位三方面改进：

- 增加网络质量监控，包括丢包率、TCP 重传率、连接失败率、跨可用区延迟等指标；
- 对核心链路增加探测任务，持续检测服务间网络连通性；
- 优化应用超时、重试、熔断、降级策略，避免网络抖动时故障扩散；
- 梳理关键依赖调用链，减少不必要的同步阻塞调用；
- 建立网络异常排查手册，明确应用、主机、网络、云厂商侧的排查顺序；
- 对核心业务增加多节点、多可用区容灾能力，降低单条链路异常的影响。

## 八、结论

本次事故主要由网络链路短时不稳定引发，应用层异常是网络抖动向上放大的结果。后续需要补齐网络监控和链路探测能力，同时提升服务在网络异常场景下的容错能力，降低类似问题对业务的影响。
