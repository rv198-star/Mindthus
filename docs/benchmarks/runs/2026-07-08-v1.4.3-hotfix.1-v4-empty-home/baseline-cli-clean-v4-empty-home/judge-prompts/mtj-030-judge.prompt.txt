# 项目风险评估

## 一、技术风险

### 风险描述  
项目涉及多个模块、接口或第三方系统，若技术方案不清晰、关键难点验证不足，可能导致开发返工、延期或上线质量不稳定。

### 主要风险点  
1. 核心功能存在技术不确定性，例如性能、兼容性、数据一致性、接口稳定性等。  
2. 第三方服务或外部系统依赖较强，接口变更、响应慢或不可用会影响开发进度。  
3. 需求频繁变更，导致架构和代码反复调整。  
4. 测试覆盖不足，问题集中暴露在联调或上线前。

### 预警信号  
- 关键技术点在开发启动后一周内仍未验证完成。  
- 接口文档频繁变更或长期不完整。  
- 同一模块多次返工。  
- 联调阶段缺陷数量明显高于预期。  
- 性能、安全、兼容性问题没有专项测试计划。

### 应对措施  
- 在正式开发前完成技术方案评审，明确架构、接口、数据流和异常处理方案。  
- 对高风险技术点先做 Demo 或 PoC 验证，不等到后期再发现问题。  
- 建立接口变更机制，所有接口调整需记录影响范围和确认人。  
- 对核心模块设置代码评审和单元测试要求。  
- 在开发中期安排一次技术风险复盘，提前暴露阻塞点。

### 责任人  
技术负责人、后端负责人、前端负责人、测试负责人。

---

## 二、市场风险

### 风险描述  
如果目标用户需求、竞品策略或政策环境发生变化，项目上线后可能出现使用率低、转化不达标或功能价值不足的问题。

### 主要风险点  
1. 项目需求来自内部判断，缺少真实用户验证。  
2. 竞品已经推出类似功能，项目差异化不足。  
3. 上线周期较长，市场窗口可能被错过。  
4. 产品定位不清，导致功能做完但用户感知价值有限。

### 预警信号  
- 需求评审中无法明确目标用户和使用场景。  
- 没有竞品分析或用户访谈记录。  
- 核心指标只写“提升体验”，没有量化目标。  
- 项目过程中频繁出现“这个功能到底给谁用”的争议。  
- 销售、运营或客服反馈与产品方案不一致。

### 应对措施  
- 在立项阶段明确目标用户、核心场景和预期指标。  
- 完成至少一次竞品分析，说明本项目的差异点。  
- 对关键功能进行用户访谈、原型验证或小范围试用。  
- 设置上线后的观察指标，例如使用率、留存率、转化率、投诉率、工单量等。  
- 如果市场变化导致原目标失效，应触发需求复审，而不是继续按原计划开发。

### 责任人  
产品负责人、市场负责人、运营负责人。

---

## 三、人员风险

### 风险描述  
项目成员若存在资源冲突、职责不清、关键人员依赖过高等问题，可能导致沟通成本上升、任务延期或交付质量下降。

### 主要风险点  
1. 关键模块依赖少数核心人员，存在单点风险。  
2. 成员同时参与多个项目，实际投入时间不足。  
3. 产品、研发、测试之间职责边界不清。  
4. 决策链路过长，问题无法及时拍板。  
5. 新成员或外包成员对业务背景不熟悉。

### 预警信号  
- 任务多次延期但没有明确原因。  
- 会议结论不清，问题反复讨论。  
- 关键人员请假或离职后无人能接手。  
- 测试阶段发现大量需求理解偏差。  
- 项目计划中没有预留评审、联调、修复时间。

### 应对措施  
- 明确项目角色：产品、技术、测试、设计、运营分别负责什么。  
- 对关键模块设置主备负责人，避免单人依赖。  
- 每周进行一次项目进度同步，重点跟踪阻塞事项，而不是只汇报完成情况。  
- 所有重要决策形成记录，包括决策内容、责任人和截止时间。  
- 对资源冲突提前升级，由项目负责人协调优先级。

### 责任人  
项目经理、部门负责人、各模块负责人。

---

## 四、交付风险

### 风险描述  
如果项目周期过紧、范围控制不当或验收标准不清，可能导致上线延期、带病上线或上线后频繁补救。

### 主要风险点  
1. 项目排期没有考虑需求评审、设计、开发、联调、测试、灰度和回滚。  
2. 需求范围持续扩大，影响原定交付时间。  
3. 验收标准不明确，导致上线前反复争议。  
4. 上线方案、回滚方案和应急预案不完善。  
5. 缺陷修复时间不足，测试被压缩。

### 预警信号  
- 项目进度连续两周低于计划。  
- 新增需求未评估影响就直接加入当前版本。  
- 上线前仍有核心流程未通过测试。  
- 缺少灰度发布、监控和回滚方案。  
- 验收人、验收标准、上线条件不明确。

### 应对措施  
- 制定分阶段里程碑：需求冻结、开发完成、联调完成、测试完成、灰度上线、正式上线。  
- 建立需求变更控制，新增需求必须评估对时间、成本和质量的影响。  
- 明确上线准入条件，例如核心流程通过率、严重缺陷数量、性能指标、回滚方案等。  
- 对非核心需求设置优先级，必要时延期到下一版本。  
- 上线前组织发布评审，确认风险、负责人和应急联系人。

### 责任人  
项目经理、产品负责人、技术负责人、测试负责人、运维负责人。
