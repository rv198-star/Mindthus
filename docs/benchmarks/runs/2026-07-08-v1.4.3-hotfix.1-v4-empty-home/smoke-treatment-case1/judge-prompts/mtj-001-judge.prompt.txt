可以先按“证据驱动建索引”，不要直接全库补索引。P99 超标通常可能是索引缺失、索引选择差、慢 SQL 参数分布倾斜、锁等待、连接池或缓存击穿共同导致。建索引方案如下。

**1. 先锁定 P99 贡献 SQL**

按接口维度拉取最近 1-3 小时的慢查询，优先排序：

```sql
-- PostgreSQL 示例
SELECT
  query,
  calls,
  mean_exec_time,
  max_exec_time,
  rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

重点看：

- P99 时段出现频率最高的 SQL
- 单次耗时高的 SQL
- 扫描行数远大于返回行数的 SQL
- `WHERE / JOIN / ORDER BY / GROUP BY` 中没有合适索引的字段

**2. 建索引优先级**

优先建这几类索引：

**高优先级：接口主路径查询**

例如：

```sql
SELECT *
FROM orders
WHERE user_id = ?
  AND status = ?
ORDER BY created_at DESC
LIMIT 20;
```

推荐：

```sql
CREATE INDEX CONCURRENTLY idx_orders_user_status_created
ON orders (user_id, status, created_at DESC);
```

原则：

- 等值过滤字段放前面：`user_id`, `status`
- 排序字段放后面：`created_at DESC`
- 和查询顺序匹配，避免额外排序

**高优先级：JOIN 外键字段**

例如：

```sql
SELECT *
FROM order_items
WHERE order_id = ?;
```

推荐：

```sql
CREATE INDEX CONCURRENTLY idx_order_items_order_id
ON order_items (order_id);
```

所有高频 JOIN 字段都要检查，尤其是子表外键。

**高优先级：状态 + 时间范围查询**

例如：

```sql
SELECT *
FROM tasks
WHERE status = 'pending'
  AND created_at < ?
ORDER BY created_at ASC
LIMIT 100;
```

推荐：

```sql
CREATE INDEX CONCURRENTLY idx_tasks_status_created
ON tasks (status, created_at);
```

如果某个状态占比很低，可以用部分索引：

```sql
CREATE INDEX CONCURRENTLY idx_tasks_pending_created
ON tasks (created_at)
WHERE status = 'pending';
```

**3. 避免乱建的索引**

先不要建：

```sql
CREATE INDEX ON table_name (low_cardinality_column);
```

比如单独给 `status`, `type`, `is_deleted` 建索引，通常收益有限，除非是部分索引或组合索引的一部分。

也不要为每个字段都建索引。索引会增加：

- 写入成本
- 更新成本
- 磁盘占用
- 查询优化器选择复杂度

**4. 推荐执行顺序**

1. 取 P99 时段 top 10 慢 SQL。
2. 对每条 SQL 跑 `EXPLAIN ANALYZE`。
3. 优先处理出现 `Seq Scan`、`Sort`、`Hash Join` 且扫描行数很大的查询。
4. 每次只加 1-3 个索引。
5. 用线上只读副本或灰度环境验证执行计划。
6. PostgreSQL 用 `CREATE INDEX CONCURRENTLY`，MySQL 用 online DDL 能力，避免锁表。
7. 上线后观察 P95/P99、CPU、IO、写入延迟和索引命中率。
8. 一周后清理未使用索引。

**5. 交付模板**

你可以把每个索引方案写成这样：

```text
接口：GET /api/orders
慢 SQL：按 user_id + status 查询订单，按 created_at 倒序分页
问题：扫描 orders 大量行并额外排序
索引：
CREATE INDEX CONCURRENTLY idx_orders_user_status_created
ON orders (user_id, status, created_at DESC);

预期收益：
减少扫描行数，避免排序，降低 P99 延迟

风险：
增加 orders 写入成本，需要观察 insert/update 延迟
回滚：
DROP INDEX CONCURRENTLY idx_orders_user_status_created;
```

如果你现在没有完整慢 SQL，我建议先只做两件事：  
先抓 P99 时段 top 慢 SQL，再针对每条 SQL 的 `WHERE + ORDER BY + JOIN` 设计组合索引。不要按字段名猜索引。
