可以加，但我需要先确认第三个 fallback 的边界，否则很容易把不稳定测试“盖住”而不是修好。

建议第三层兜底只处理明确的临时性失败，例如：

- 上游返回空结果或超时
- 解析失败但原始响应可用
- 前两个 fallback 都没有产出有效值
- 非核心字段缺失，可以返回降级默认值

不要兜底：

- 参数错误
- 权限错误
- schema 明显不兼容
- 业务断言失败

实现上我会按这个顺序处理：

```text
primary -> fallback_1 -> fallback_2 -> fallback_3
```

第三个 fallback 建议返回最保守、最稳定的结果，并打日志或标记来源，例如 `source: "fallback_3"`，这样测试稳定了以后也能看出是不是长期依赖兜底。
