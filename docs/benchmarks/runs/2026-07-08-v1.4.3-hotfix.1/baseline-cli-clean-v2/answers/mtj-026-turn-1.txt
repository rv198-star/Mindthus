治法是把“退款承诺”从客服 agent 的自由生成里拿出来，改成受控动作。

建议分四层做：

1. **权限隔离**
   - Agent 可以解释政策、收集信息、建议“可能符合退款条件”。
   - 但不能直接说“我们会退款”“已为你退款”“保证到账”。
   - 真正退款只能走工具调用或工单状态，而工具必须有人审或规则审。

2. **硬性输出约束**
   在系统提示里明确禁止：
   - 承诺退款
   - 承诺金额
   - 承诺到账时间
   - 暗示审批已通过

   允许话术改成：
   > 我可以帮你提交退款申请，结果需要审核后确认。

3. **工具层拦截**
   不要只靠 prompt。给退款相关工具加权限：
   - `submit_refund_request`：agent 可调用
   - `approve_refund` / `issue_refund`：agent 不可调用，必须人工或确定性规则调用
   - 所有涉及退款承诺的回复，在发送前过一个 policy checker

4. **自动检测和升级**
   对 agent 回复做发送前扫描，命中这些意图就拦截：
   - “给您退款”
   - “一定退款”
   - “已通过”
   - “多久到账”
   - “退回原支付方式”
   - 明确金额承诺

   命中后改为：
   - 阻止发送
   - 自动转人工
   - 记录样本用于回归测试

最关键的一点：**不要让模型“决定能不能退款”，只让它“发起退款流程”**。退款资格、金额、审批状态必须来自后端规则或人工审批结果，而不是模型语言生成。
