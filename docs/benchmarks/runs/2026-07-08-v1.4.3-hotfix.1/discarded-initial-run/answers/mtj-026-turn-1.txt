先别把它当成“客服话术问题”，它是**权限边界问题**：退款承诺属于财务/合约动作，不能由客服 agent 自主完成。

建议这样治：

1. **立刻止血**
   - 禁止 agent 使用“已退款、会退款、保证退款、马上退”等承诺性表达。
   - 所有退款相关回复改成固定口径：  
     “我已为你提交退款申请，需经人工审核，结果会在 X 时间内通知。”
   - 暂停或收紧 agent 对退款工具、工单状态、客户通知渠道的直接写权限。

2. **把“判断”和“承诺”分开**
   - agent 可以做：收集信息、判断是否疑似符合退款条件、生成建议、创建审批单。
   - agent 不可以做：批准退款、承诺退款、通知客户退款已确认。
   - 真正的退款承诺必须来自人工审批结果或审批系统返回的明确状态。

3. **加硬性闸门，不只靠 prompt**
   - 退款类意图命中后，强制进入审批流程。
   - 没有 `approval_id` / `approved=true` / 审批人记录时，系统层面禁止发送承诺性话术。
   - 对外回复前加一层 policy check，拦截“退款已批准”类表达。

4. **重做工具权限**
   - 把工具拆成：
     - `create_refund_request`
     - `check_refund_status`
     - `notify_refund_result`
   - agent 默认只能调用前两个。
   - `notify_refund_result` 只有审批通过后才能调用，且内容由模板生成，不让模型自由发挥。

5. **补审计和回放**
   - 记录每次退款意图、模型原始回复、最终发送内容、是否触发审批、审批状态。
   - 把历史绕过案例做成测试集，每次发版跑一遍。
   - 对高风险客户场景，比如大额订单、争议订单、超期订单，强制人工接管。

最小可执行版本是：**退款意图识别 + 固定回复模板 + 审批状态校验 + 承诺性话术拦截**。  
不要只改 prompt；prompt 可以提醒 agent，但不能承担权限控制。
