“单页应用已经过时”这个前提不一定成立。更准确的判断是：如果你们现在的 SPA 在首屏性能、SEO、分享预览、低端设备体验、权限首包、可观测性或运维复杂度上吃亏，SSR/SSG/混合渲染值得迁移；但不建议把“全站改回 SSR”当成默认目标。更稳妥的目标是：按页面价值选择 SSR、SSG、ISR 或继续 CSR。

迁移步骤可以这样排：

1. 盘点现有页面和路由

   把页面分成几类：

   - 强 SEO / 落地页 / 内容页：优先 SSR、SSG 或 ISR。
   - 登录后业务后台：多数不需要 SSR，可继续 CSR 或只做壳 SSR。
   - 高交互工作台、编辑器、图表页：谨慎迁移，SSR 收益可能低。
   - 需要分享卡片、首屏快、权限敏感的页面：适合先试点 SSR。

2. 明确迁移目标

   不要只写“改 SSR”，要写清楚目标指标：

   - 首屏加载时间降低多少。
   - SEO 收录或页面可见性改善到什么程度。
   - 服务端成本可接受范围。
   - 是否允许页面级混合渲染。
   - 是否必须保留现有交互体验。

3. 选定技术路线

   常见选择：

   - React 项目：Next.js。
   - Vue 项目：Nuxt。
   - Svelte 项目：SvelteKit。
   - 已有 Node BFF：可以做渐进式 SSR。
   - 内容站为主：优先 SSG/ISR，不一定全 SSR。

4. 先做一个试点页面

   选一个收益明确、依赖适中的页面，比如首页、详情页、列表页。

   试点要验证：

   - 服务端数据获取。
   - 登录态和权限处理。
   - 路由兼容。
   - SEO 元信息。
   - 缓存策略。
   - 部署链路。
   - 性能指标。
   - 错误监控。

5. 拆分数据获取逻辑

   SPA 里很多数据请求写在组件生命周期里，迁移时要抽出来：

   - 服务端可执行的数据请求。
   - 客户端交互后的增量请求。
   - 需要鉴权的请求。
   - 可缓存的公开请求。
   - 不能在服务端暴露的浏览器专属逻辑。

6. 处理浏览器专属代码

   迁移 SSR 时重点排查：

   - `window`
   - `document`
   - `localStorage`
   - `sessionStorage`
   - `navigator`
   - DOM 测量
   - 第三方 SDK
   - 图表、编辑器、地图组件

   这些代码要延迟到客户端执行，或拆成 client-only 组件。

7. 设计缓存和失效策略

   SSR 不是越实时越好。要按页面选择：

   - 静态内容：构建时生成。
   - 低频变化内容：ISR / 页面缓存。
   - 高频个性化内容：SSR 或客户端请求。
   - 登录后数据：通常不做公共缓存。
   - 商品、文章、配置类数据：需要明确失效机制。

8. 迁移路由和页面壳

   保持 URL 尽量不变，避免 SEO 和外链损失。

   需要处理：

   - 旧路由映射。
   - 301/302 重定向。
   - 动态路由。
   - 404 页面。
   - 错误页面。
   - canonical URL。
   - sitemap 和 robots。

9. 迁移状态管理

   原 SPA 的全局状态通常需要重新分层：

   - 服务端初始数据。
   - 客户端交互状态。
   - 登录用户状态。
   - 跨页面缓存。
   - URL 查询参数状态。

   不建议把所有客户端状态硬塞到 SSR 里。

10. 建立双轨发布

   不要一次性全站切换。

   推荐顺序：

   - 新框架接入。
   - 单页面试点。
   - 小流量灰度。
   - 指标对比。
   - 逐批迁移高收益页面。
   - 最后处理低收益、高风险页面。

11. 补齐测试和监控

   至少覆盖：

   - 服务端渲染是否成功。
   - hydration 是否报错。
   - 首屏性能。
   - SEO 元信息。
   - 登录态。
   - 路由跳转。
   - 错误边界。
   - 服务端接口失败。
   - 缓存命中率。
   - Node 服务资源占用。

12. 制定回滚方案

   每一批迁移都要能回滚：

   - 按路由切回旧 SPA。
   - 保留旧构建产物。
   - 配置层控制流量。
   - CDN / 网关支持快速切换。
   - 数据接口保持兼容一段时间。

建议最终迁移策略不是“全站 SSR”，而是：

- 内容、营销、公开详情页：优先 SSR/SSG/ISR。
- 登录后业务系统：保留 CSR 或局部 SSR。
- 高交互复杂页面：除非首屏或权限收益很明确，否则不要急着迁。

这样迁移成本更低，风险也更可控。
