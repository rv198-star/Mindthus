先别只补一句“退款必须审批”的 prompt，这类问题通常要从权限、流程、证据三层一起治。

核心处理：

1. **把“承诺退款”从 agent 权限里拿掉**
   - agent 可以说：`我已为你提交退款申请，会由专员审核。`
   - 不能说：`我们会给你退款`、`退款已批准`、`肯定退`。
   - 退款承诺必须由后端权限控制，不靠模型自觉。

2. **把退款动作拆成两类**
   - `建议退款`：agent 可生成理由、金额、订单证据。
   - `批准退款 / 对客户承诺退款`：必须走人工或规则引擎审批。

3. **加工具层硬限制**
   - 如果 agent 有调用 CRM、工单、IM 的能力，工具 schema 里不要给它“批准退款”或“发送承诺话术”的直通能力。
   - 发送客户消息前做一次 policy check：检测“退款已批准、一定退款、马上到账”等承诺语义，命中则拦截或转人工。

4. **改对客话术模板**
   允许：
   > 我已记录你的退款诉求，并提交给审核团队。审核完成后会通知你结果。

   禁止：
   > 我们会给你退款。  
   > 退款已经通过。  
   > 这个肯定可以退。

5. **建立事故回放**
   每次绕过审批都记录：
   - 触发用户话术
   - agent 原始回复
   - 是否调用了工具
   - 哪个规则没拦住
   - 最终损失金额或投诉后果

6. **用小测试集回归**
   专门测这些高风险场景：
   - 客户威胁投诉
   - 客户说“上次客服答应了”
   - 客户情绪激烈
   - 订单明显有问题但政策不确定
   - 客户要求“你现在就确认能不能退”

一句话：**不要让模型“决定自己是否该升级人工”，而是让系统在退款承诺这个边界上强制接管。** Agent 负责收集事实和生成建议，审批和承诺必须由流程或人工控制。
