不能因为“SPA 过时”就直接全站改 SSR；正确顺序是先确认网站要完成的工作、现有约束和证据，再决定哪些页面值得迁移到 SSR、SSG、ISR 或继续保留客户端渲染。

建议迁移步骤：

1. **先定义目标**
   明确这次迁移到底为了解决什么：SEO、首屏性能、转化率、可访问性、稳定性、运维成本，还是内容分发效率。  
   没有目标，不要启动全站迁移。

2. **盘点页面类型**
   把页面分成几类：
   - 营销页、官网页、内容页、商品详情页：优先考虑 SSR/SSG/ISR。
   - 登录后后台、编辑器、仪表盘、复杂交互页：通常继续 SPA 或混合渲染。
   - 高实时性页面：重点看缓存策略和数据新鲜度，不是盲目 SSR。

3. **收集现状证据**
   至少看这些数据：
   - 首屏 LCP、TTFB、INP、CLS。
   - 搜索收录和自然流量。
   - 页面转化率。
   - JS bundle 大小。
   - 服务端成本和接口耗时。
   - 线上错误率。

4. **选择迁移策略**
   不建议“一刀切全站 SSR”。更稳妥的是：
   - 静态内容用 SSG。
   - 需要定期更新的内容用 ISR 或增量生成。
   - 强 SEO 且数据动态的页面用 SSR。
   - 登录后重交互页面保留 CSR/SPA。
   - 公共布局、鉴权、数据加载逐步抽象。

5. **先做试点页面**
   选 1-3 个高价值页面验证，比如首页、详情页、落地页。  
   对比迁移前后的性能、SEO、转化率和服务端成本。

6. **设计基础设施**
   迁移前要确定：
   - 路由方案。
   - 数据获取方式。
   - 缓存层。
   - CDN 策略。
   - 鉴权和会话处理。
   - 错误监控。
   - 灰度发布和回滚机制。

7. **拆分共享代码**
   把现在 SPA 里的逻辑拆成：
   - 纯 UI 组件。
   - 客户端交互组件。
   - 服务端数据加载逻辑。
   - 通用 API 层。
   - SEO metadata 生成逻辑。

8. **逐页迁移**
   按收益排序迁移，而不是按目录顺序：
   - SEO/流量高的页面优先。
   - 首屏慢的页面优先。
   - 业务价值高的页面优先。
   - 复杂交互页最后处理。

9. **建立双轨验证**
   每迁一个页面都要验证：
   - 页面功能一致。
   - SEO 标签正确。
   - 首屏性能改善。
   - 缓存没有脏数据。
   - 服务端压力可接受。
   - 客户端交互没有退化。

10. **最后再决定是否“全站”**
   只有当证据显示 SSR/SSG 对多数页面都有收益，且成本可控，才推进全站级改造。否则应保持混合架构。

一句话结论：不要把“从 SPA 改回 SSR”当成目标，目标应该是提升可被验证的业务和体验指标；SSR 只是其中一种实现手段。
