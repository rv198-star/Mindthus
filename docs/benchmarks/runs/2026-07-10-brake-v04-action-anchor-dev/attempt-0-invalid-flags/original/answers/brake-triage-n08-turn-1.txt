建议把“售后规则入口”从页面提示改成一个统一的规则决策入口：用户不是先选退款、换货、赠品，而是先回答“我遇到什么问题”，系统再给可执行路径。

**方案核心**

把入口做成三层：

1. **售后规则首页**
   - 展示用户最常遇到的问题：
     - 我要退款
     - 我要换货
     - 少发 / 漏发 / 赠品未到
     - 商品破损 / 质量问题
     - 物流异常
     - 超时未处理
   - 每个入口不要写长规则，只写“适用场景 + 下一步”。

2. **规则判断页**
   - 用户进入后，用 2-4 个问题判断：
     - 是否已签收
     - 是否超过售后时效
     - 是否拆封 / 使用
     - 是否有凭证
     - 是否涉及赠品、活动、套装
   - 判断后给出明确结果：
     - 可申请
     - 需补充凭证
     - 不支持但可联系客服
     - 建议走另一类售后

3. **行动页**
   - 给用户一个明确动作：
     - 立即申请退款
     - 申请换货
     - 补传照片
     - 联系客服
     - 查看处理进度
   - 同时展示关键规则摘要，而不是整篇规则。

**入口设计**

不要再把规则散落在退款页、换货页、赠品页。建议统一放在这些位置：

- 订单详情页：`申请售后` 主入口
- 售后列表页：`售后规则 / 我该怎么处理`
- 客服页：`常见售后问题`
- 活动赠品页：只保留轻提示，点击跳转到统一入口
- 退款、换货、赠品页：只保留规则结果引用，不再各自维护一套说明

**信息结构**

规则中心建议按“用户问题”组织，而不是按公司内部规则组织：

- 退款规则
- 换货规则
- 赠品 / 活动权益规则
- 物流异常规则
- 质量问题规则
- 售后时效规则
- 凭证要求
- 不支持售后的情况

每条规则都要有固定结构：

- 适用场景
- 是否支持
- 需要什么材料
- 处理时效
- 用户下一步
- 例外情况

**改造顺序**

1. 先整理现有退款、换货、赠品页的所有补丁内容，合并成一份规则库。
2. 把重复、冲突、模糊的规则清掉，保留一个唯一口径。
3. 设计统一入口和问题分流。
4. 先接订单详情页和售后列表页，不急着全站替换。
5. 观察用户点击、转人工、申请失败率，再决定是否下掉旧页面提示。

**判断标准**

这个入口做好后，应该解决三个问题：

- 用户知道自己该走退款、换货还是赠品补发。
- 客服不需要反复解释同一批规则。
- 规则修改只改一处，不再到处贴补丁。

一句话：不要继续给页面打补丁，改成“统一规则库 + 场景化分流 + 明确下一步”的售后决策入口。
