# Mindthus 中文说明书

Mindthus 是一套把方法论、边界和行动控制整理成可复用技能包的项目。

它的目标不是把文档写得更满，而是让 agent 更快判断：

- 这是什么问题
- 该用哪种方法
- 哪些边界不能越
- 哪些动作该继续，哪些该停下

## Mindthus 是什么

Mindthus 是个人方法论与 agent 构建项目，核心产物是：

- `AGENTS.md`：项目级姿态和边界
- `skills/*/SKILL.md`：独立技能入口
- `skills/*/resources/`：长方法资源和运行支撑
- `docs/methodologies/`：跨技能的方法说明

## Mindthus 能做什么

- 让 agent 先判断问题类型，再选方法
- 把模糊问题变成可复述、可定位、可证伪的问题
- 把复杂任务变成可执行、可验证、可回退的步骤
- 把长任务的继续/停止、证据、日志和归档分开
- 把局部修补螺旋挡在控制边界之外

## Mindthus 如何做到

### 1. 项目姿态先行

`AGENTS.md` 先规定默认姿态：先搞清楚情况，再行动。  
这让后面的技能不是随便调用，而是按问题类型路由。

### 2. 技能分工明确

每个 `SKILL.md` 只负责一个明确方法：

- `using-mindthus` 负责路由与前置校准
- `SELA` 负责整体与局部的系统判断
- `3L5S` 负责问题发现、定义和落地
- `EDSP` 负责结构判断
- `WAE` 负责控制权边界
- `tplan` 负责 Mission 级任务运行
- `Anti-Spiral` 负责防止局部修补螺旋

### 3. 资源和入口分层

短入口留在 `SKILL.md`，长解释放在 `resources/`。  
这样既能快速加载，也能保留完整方法背景，不把主路径淹没在补丁里。

### 4. 测试把格式固定住

仓库里的测试不是装饰，它们在固定这些约束：

- `SKILL.md` 必须按分层组织
- 项目说明必须包含方法分层纪律
- 关键方法资源必须存在并包含核心措辞

## 使用场景

### 重大趋势判断

当局部优势真实，但系统级费效比已经明显倾斜时，用 `SELA`。

### 混乱问题拆解

当问题本身还说不清时，用 `3L5S` 先把问题定义出来，再谈执行。

### 结构判断

当 A/B 都像对、但命题本身可能有问题时，用 `EDSP`。

### 控制边界

当不确定谁控制流程、谁控制判断、谁控制证据时，用 `WAE`。

### 长任务运行

当任务需要持续推进、记录证据和处理停止条件时，用 `tplan`。

### 防止局部修补螺旋

当同一局部对象被反复处理、补丁不断加层时，用 `Anti-Spiral` 先停下来。

## SELA 的哲学底座

SELA 不是要求把所有局部都压平到同一个效率目标上。更准确地说，它要求整体给方向，局部给真实性，边界给不可逾越的约束。

真正可落地的不是一次性静态最优，而是可逆、可观察、可修正的持续逼近：

- `可逆`：错了能退
- `可观察`：能及时看见偏差
- `可修正`：反馈来了能改

因此，SELA 反对把“长期方向正确”误写成“现在立刻全切”。整体最优和局部最优的关系，不是消灭局部，而是让局部保留校准系统的能力。
