# Private Repository / Public Skills Release Boundary

## Status

Internal maintainer note for release architecture. This document records the package
boundary that lets the repository keep private planning material while publishing a
clean, free, installable Mindthus skills artifact.

v0.6.2 should close the design boundary. It does not need to finish every package
hardening item before release.

## Core Decision

Mindthus should use an allowlist-generated release pack as the initial public
distribution strategy.

The full repository remains the development surface. The public release is a generated
artifact built from selected source classes by `scripts/build-release-pack.py`. Public
users should be able to install from the release pack without depending on a checkout of
the full repository.

This boundary is a release architecture rule, not a skill behavior rule. It should not
change `SELA`, `3L5S`, `EDSP`, `WAE`, `TVG`, `tplan`, or `using-mindthus` semantics.

## Included file classes

The public skills release pack may include:

- skill entrypoints: `skills/*/SKILL.md`
- skill runtime support that is needed by those entrypoints: `resources/`, `templates/`,
  and deterministic `scripts/` under a packaged skill
- public methodology references under `docs/methodologies/`
- public install and usage guidance appropriate for packaged users
- platform-specific packaging metadata for OpenCode, Claude Code marketplace, and Codex
- public release metadata such as `README.md`, `CHANGELOG.md`, `LICENSE`, manifest files,
  and checksums when those are generated

The initial builder already creates separate platform layouts:

- OpenCode: `.opencode/skills/mindthus/<skill>/SKILL.md`
- Claude Code marketplace: `claude-code/.claude-plugin/marketplace.json` plus
  `claude-code/claude-plugin/.claude-plugin/plugin.json`
- Codex: `codex/skills/mindthus/<skill>/SKILL.md`

## Excluded file classes

The public skills release pack must exclude:

- `docs/internal/`
- `docs/superpowers/`
- private specs, plans, issue drafts, and maintainer-only notes
- raw evaluation traces that are not intentionally public release evidence
- tests unless a later release explicitly packages public contract tests
- untracked drafts
- local worktrees, local absolute paths, runtime logs, caches, and temporary artifacts
- secrets, tokens, credentials, `.env` style files, and private endpoint references

Exclusion should be enforced by building from allowlisted source roots rather than by
copying the repository and deleting private paths afterward.

## Cleanliness validation

A release pack is clean only if validation can prove the artifact boundary, not merely
that the current maintainer remembered to remove sensitive files.

Required validation direction:

- reject forbidden paths such as `docs/internal/` and `docs/superpowers/`
- reject symlinks that escape the staged package root
- verify every packaged skill has `SKILL.md` with valid frontmatter, `name`, and
  `description`
- verify references from packaged skills resolve inside the package or to intentional
  public documentation
- fail on likely secrets, tokens, private keys, credential files, local absolute paths,
  and maintainer-only URLs
- fail on suspicious private phrases unless they are explicitly allowed in public docs
- verify platform layouts for OpenCode, Claude Code marketplace, and Codex
- verify manifest file lists and hashes once manifest generation exists

The current `tests.test_packaging_docs` coverage checks the platform layout generated by
`scripts/build-release-pack.py`. A fuller validator should become a separate release
hardening task.

## Installation boundary

`scripts/install-skills.sh remains the development checkout installer`. It can continue
to install directly from a local repository checkout for maintainers.

The release installer must operate on the sanitized package. It should copy from the
release pack by default, not symlink back to the repository, because a public package must
not require access to private repository content.

Expected install targets:

- OpenCode user install: `~/.config/opencode/skills/mindthus/<skill>/SKILL.md`
- Claude Code user install: install through a Claude Code marketplace/plugin package,
  not a guessed direct `~/.claude/skills` copy path
- Codex user install: `${CODEX_HOME:-~/.codex}/skills/mindthus/<skill>/SKILL.md`
- shared Agent Skills compatibility: `~/.agents/skills/mindthus/<skill>/SKILL.md`

The installer should support dry-run, list, target override, force, and non-overwrite by
default. It must not execute bundled skill runtime scripts during installation.

## Current v0.6.2 scope

v0.6.2 should preserve the existing checkout install flow and document the release
boundary clearly enough that future packaging work has a fixed target.

Do not make repository privacy, license change, or destructive cleanup part of v0.6.2.

v0.6.2 can ship with:

- the existing `scripts/build-release-pack.py` platform pack builder
- this design boundary
- contract tests that prevent the boundary from disappearing
- follow-up work orders for deeper validation and release asset generation

## Follow-up work orders

1. Add a full public package validator.
   Validate forbidden paths, forbidden phrases, likely secrets, symlink escape, internal
   references, skill frontmatter, platform layouts, and script syntax.

2. Generate canonical GitHub Release assets.
   Produce `mindthus-skills-<version>.tar.gz`, `mindthus-skills-<version>.zip`,
   `manifest.json`, and `SHA256SUMS`.

3. Add install from the release pack.
   Provide a package-local installer that supports OpenCode, Claude Code marketplace,
   Codex, and shared Agent Skills compatibility targets.

4. Decide LICENSE handling for the public skills package.
   Keep the skills open and free, but do not silently infer license terms from future
   repository privacy changes.

5. Add platform smoke tests for release artifacts.
   Check OpenCode discovery and invocation, Claude Code marketplace validation and
   invocation, and Codex skill registry visibility and invocation.

6. Decide whether a separate public repository is needed.
   Keep the current generated-pack strategy unless public distribution, contribution, or
   marketplace requirements make a split repository materially cleaner.
