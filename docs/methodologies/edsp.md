# EDSP / Extreme Deduction + Scenario Projection

## 这是什么

EDSP 是 `Extreme Deduction + Scenario Projection`，一套用于定性结构判断的方法。

它适合处理那种“怎么说都像有道理”的问题：A 和 B 都对，边界不清，原则在具体场景里摇摆，或者命题本身可能就是错的。EDSP 不急着给答案，而是先把关键变量推到极端，建立一个稳定的结构坐标系，再把现实场景投进去看。

`Extreme Deduction` 负责把模糊命题压成结构；`Scenario Projection` 负责在结构稳定后处理具体场景。两者顺序很重要：如果坐标系还没建好，场景投影只会把直觉包装得更漂亮。

## 解决什么问题

EDSP 解决的是结构判断里的犹豫和伪二选一。

常见场景包括：

- A/B 都像对，但争论一直停留在偏好层。
- 问题被写成二选一，但真实结构可能不是二选一。
- 某个原则看似正确，但落到不同场景时结论相反。
- 一个命题听起来合理，却可能把变量混在一起。
- 趋势判断、伦理边界、产品取舍或工程原则需要更清楚的坐标系。

它对 agent 特别有用，因为 agent 很容易把双方理由都讲得通顺，然后停在“取决于上下文”。EDSP 要求先定义上下文的结构，而不是把“看情况”当结论。

## 核心判断

EDSP 的核心判断是：先推极端，再读结构；先定坐标，再投场景。

在 `Extreme Deduction` 中，先找关键变量，然后把它们推到极端。极端不是为了制造夸张，而是为了让变量的真实作用显形。例如把成本推到接近零，把延迟推到接近无限，把错误代价推到不可逆，把用户规模推到很大，原本模糊的判断往往会显出几种离散结构。

当结构坐标稳定后，才进入 `Scenario Projection`。这时具体场景不再是散乱例子，而是被投到坐标系中的点。你可以看它靠近哪个边界、受哪个变量主导、需要哪类证据才能继续判断。

EDSP 的关键成果不一定是一个答案，而是一个更好的问题形状。

## 怎么用

一个轻量 EDSP 可以这样做：

1. 写出原命题，尤其是那个让人摇摆的判断。
2. 找出 2-4 个关键变量，不要一上来列十几个因素。
3. 把每个变量推到极端，看结论是否反转。
4. 合并极端结果，形成少量结构坐标。
5. 判断现实场景更靠近哪个坐标区域。
6. 只有坐标稳定后，才给出场景建议。

例如一个判断如果在“小规模、低风险、可逆”时成立，但在“大规模、高风险、不可逆”时反转，那么问题的关键可能不是 A/B 谁更好，而是规模、风险和可逆性在控制结论。

输出可以很短：`结构变量 -> 极端推演 -> 坐标系 -> 当前场景落点 -> 建议动作`。

## 常见误用

最常见的误用，是把 EDSP 当作证据替代品。结构判断能告诉你应该看哪些变量，不能替你证明事实已经成立。

第二种误用，是先做 Scenario Projection，再回头补 Extreme Deduction。这样容易把自己已有偏好投射到场景里，看起来很细，实际坐标不稳。

第三种误用，是把漂亮的二维矩阵当成答案。坐标系只是为了减少混乱，如果变量选错、极端推演不成立，矩阵越漂亮越危险。

## 边界

EDSP 不替代领域研究、运行时证明、用户访谈、法律判断或 stakeholder 决策。如果真正缺的是事实，就先补事实。

它也不适合低不确定性任务。确定性的格式化、安装、测试、机械重构，不需要用 EDSP 包装。

如果问题卡在“怎么拆执行”，转 `3L5S`。如果卡在“谁控制这段工作”，转 `WAE`。如果卡在“长期系统效率是否压过局部优势”，转 `SELA`。

## 与其他方法的关系

- `3L5S` 提供问题处理主干，EDSP 处理其中的结构摇摆。
- `SELA` 可以借 EDSP 判断局部优势和系统效率的变量边界。
- `WAE` 约束 EDSP 输出应该由判断、流程还是证据接管。
- `tplan` 可以把 EDSP 变成 Mission 中的 decision hook。
