# TVG-Profile 案例一：`plain-sharp-skill-intro`

## 这页讲什么

这个案例是我们给 TVG-Profile 准备的一条轻量起步线。我们想说明一件事：Profile 不一定要从脚本、赛马、
资源包开始，很多时候只要先把“什么叫好的入口文案”说清楚，TVG 就已经能明显变得更有方向。

这页不是在复盘一个实验，而是在告诉你：如果你想把方法页、skill 介绍页、说明页写得更像产品前门，
可以从这个案例开始。

对应资源：`skills/tvg/resources/value-profiles/plain-sharp-skill-intro.md`

## 我们为什么做这个案例

我们在整理 Mindthus 方法页和 skill 入口时，反复遇到一个问题：文档明明写全了，读者还是抓不住
“这东西现在对我有什么用”。它的问题不是信息太少，而是入口判断太散，像说明书，不像前门。

所以我们做了 `plain-sharp-skill-intro`。它要约束的不是“更会写文案”，而是让一句话先点破本质，再用一个普通动作或普通错误把判断面照亮，
最后把边界说清楚。我们的目标不是宣传腔，而是一个可靠、好用、第一次看也能懂的产品入口。

## 我们在这个案例里用了哪些层

- `value_semantics`：规定什么叫“说清楚”。例如一句话道破本质、短句承载关键判断、例子只解释一个动作、边界要诚实。
- `realization_surface`：把“好不好”落到可检查单位上，比如 quotable first sentence、essence sentence、sentence function、when-to-use cue、boundary sentence。
- `gain_policy`：告诉 TVG 应该优先压缩什么、替换什么、拆开什么。它鼓励把抽象夸奖改成具体失败，把长句拆成功能明确的短句。
- `runtime_support`：没有。这很重要，因为它说明 Profile 并不等于脚本包。

## 这个案例想让你直接看到什么

我们最想借这个案例说明的，不是语气更利落，而是入口文案到底应该完成哪些工作：

- 第一件事，是让读者不靠缩写也能复述这个 skill 的真实 job。
- 第二件事，是让高信息密度来自短句里的判断，而不是长句塞满术语。
- 第三件事，是给每句话分配职能。句子不是都在“解释”，而是分别承担 essence、reason、example、boundary、use condition。
- 第四件事，是强制例子回到一个普通动作。这样读者记住的不是抽象价值，而是“我在什么场景下该想到它”。

## 你可以怎么用这个案例

如果你正在写方法介绍、skill 前言、README 的入口段，或者任何“用户第一次接触你这个东西”的文字，
这个案例基本都能直接借过去。我们希望它带给你的，不是某种统一腔调，而是一套更稳的入口纪律。

第一，Profile 不必从“复杂”开始。你不需要先上脚本、上评分、上资源包，才算真正用了 Profile。只要先把“什么算好”
和“好应该出现在哪里”说清楚，TVG 就已经从泛泛扩写，进入有方向的强化。

第二，Profile 也不必只服务风格任务。很多人一听 Profile，会先想到品牌语气、导演感、视觉风格。但这个案例说明，
一个普通的技能介绍页，同样值得有自己的价值语义。

## 我们刻意没有做什么

这个案例没有做脚本 lint，没有做运行时支持，没有做多轮结构竞争，也没有把 skill 介绍写成迷你方法论。
我们故意把它停在这里，是想保留一条足够轻、又确实有用的起步路线。要是你的问题已经变成事实边界不清、结构分歧很大，
或者下游需要机器可检查的字段完整性，那就该上更重的包，而不是继续逼这个案例加层。

## 如果你要迁到自己的项目里

如果你也想给别的文档类型做一个轻量 Profile，我们最建议你借的是这套骨架，而不是表面措辞：

1. 先定义一句话本质，不让读者先学术语。
2. 再指定一两个可检查单位，确保“好”不是全凭感觉。
3. 最后写明哪些加深动作通常有用，哪些只是让文档更像完成品。

这三步做完，很多原本会滑向“越写越全”的入口文档，通常就会开始变得更短，也更准，更像一个真的前门。
