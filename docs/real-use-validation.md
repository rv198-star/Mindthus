# Real-Use Validation / 真实使用验证

## 目标

Mindthus 的下一阶段不是继续对公开题库调分，而是回答一个更朴素的问题：它在正常工作里
是否真的改变了判断、减少了返工，还是只增加了流程负担。

启动状态：采集协议已启用；仓库不预填或伪造真实任务记录。第一轮观察窗口为 10–20 个
自然发生的任务。

## 纳入条件

一条记录只有同时满足以下条件才进入真实使用分母：

- 任务来自正常工作需求，不是为测试 Mindthus 临时编写或改写的题面。
- 记录的是一次完整任务，而不是为了凑样本拆出的多个轮次。
- 场景摘要已经脱敏，不包含客户数据、密钥、完整私有 prompt 或不可公开产物。
- 明确记录调用方式：显式 router、显式具体 skill，或 host 的 best-effort 自动唤起。
- 在结果已可观察时再记录；暂时看不出效果可以填 `unknown`，不强行打分。

公开 benchmark、fixture、影子题和为复现 bug 生成的变体仍可进入 evaluation log，但不能
混入这 10–20 个真实任务。

## 每条记录看什么

使用 `scripts/log-fidelity-usage.py` 记录：

- `decision_changed`：是否改变了原本会采取的判断或行动。
- `rework_reduced`：是否减少了后续返工、来回确认或错误路径投入。
- `overhead_level`：额外阅读、步骤和等待成本。
- `harm_observed`：是否导致更差判断、延误、过度介入或不必要阻断。
- `mechanism`：用一句脱敏的话描述真正起作用或失效的机制。

真实任务不要求 baseline 或数值 rubric。可以有分数，但不能为了“可量化”硬造一把尺子。

## 立项纪律

- 一个普通的单次失败只记账，不立优化 issue。
- 同一机制在至少 2 个相互独立的真实任务中重复，并造成可观察成本时，才进入立项审查。
- 单次高严重度安全、隐私或不可逆损害可以立即立项，但必须附真实 evidence。
- 第 10 条和第 20 条记录各做一次聚合 review；聚合看正收益、负收益和负担，不只看均分。
- 没有重复机制就不改 prompt、matcher、runtime gate 或方法正文。

## 与公开基准的关系

50 场景公开基准保留为 release-level regression surface：只在发布前、跨方法合同变化或真实
任务暴露重复机制后重跑。它不再承担日常优化导航，也不能替代独立影子证据或真实使用反馈。

真实使用记录能决定“下一步值得修什么”；公开基准只能帮助确认“修完有没有破坏已知行为”。
