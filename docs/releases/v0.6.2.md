# Mindthus v0.6.2 发布日志

发布日期：2026-05-31

## 版本定位

v0.6.2 是一次小版本发布，但它解决的是 TVG 的核心方向问题：TVG 不应该只把
AI 产物越做越厚，而应该先判断厚度是否足够，再决定当前动作是继续加深、提炼、
压缩、校准，还是停止。

这版也补齐了公开技能包发布边界：哪些内容可以进入公开 release pack，哪些内容必须
留在私有或维护者上下文里，后续发布硬化应该沿什么方向继续做。

## 这次解决了什么

之前 TVG 的主路径容易给 agent 一个隐含暗示：如果产物浅，就继续补层、补边界、
补结构。这个方向能修复薄产物，但也会制造另一个问题：产物变长了，价值密度却没有
提高。

v0.6.2 把 TVG 调整为价值增益循环：

- 厚度是价值前提：没有足够约束、场景、取舍、失败路径和证据边界，就不能急着压缩。
- 洞察是核心收益：目标是产出“没这么想过，但站得住”的观点，而不是把模板填满。
- 密度是出口质量：厚度足够后，应该把内容提炼成更有判断力、更可用的交付形态。

## 主要变化

### TVG：从扩厚度转向价值增益

- 新增厚度闸门：厚度不足时不得提前压缩，不得用“高密度表达”掩盖思考底座不足。
- 新增状态驱动动作：根据产物状态选择加深、定向补厚、提炼、压缩增强、校准或退出。
- 新增 grounded novelty / grounded stretch 边界：允许有根基的大胆设想，但不允许空想。
- 调整 claim calibration 和 handoff readiness：默认作为后段警惕提醒，不提前压制思考张力。
- 增加出口分级提炼：厚度足够后，再按输出档位调整最终表达。

### 输出档位：只改变交付倾斜，不改变内部主线

新增三个输出档位：

- `insight_dense`：更强调洞察密度和判断锋利度，但不能在厚度不足时提前压缩。
- `balanced`：保持默认平衡，避免为了中庸而回避判断。
- `coverage_rich`：保留更多场景、边界、评审项和交接结构，但不能放任低价值扩写。

其中 `coverage_rich` 的定位最接近旧版 TVG 的优势：保留有用厚度，同时用新版机制
防止内容变成“又长又散”。

### 发布边界：补齐公开技能包发布设计

#9 已补齐并关闭。本版新增内部维护者文档：

- `docs/internal/public-skills-release-boundary.md`

它明确了：

- 公开技能包可以包含哪些文件类别；
- `docs/internal/`、`docs/superpowers/`、私有计划、原始评测痕迹和本地草稿不能进入公开包；
- 当前采用 allowlist 方式生成 release pack；
- `scripts/install-skills.sh` 继续作为开发 checkout 安装器；
- 后续 release installer 必须从清理后的 release pack 安装，而不是依赖私有仓库 checkout。

## 验收结果

TVG 旧版 vs 新版 A/B 复测结果：

- `coverage_rich` 产品评审：新版胜，`9 vs 7`。
- `insight_dense` 执行摘要：新版胜，`9 vs 7`。
- `balanced` 运营规则：新版胜，`49/50 vs 33/50`。

这个结果不应该被解读为“新版任何时候都天然优于旧版”。更准确的判断是：新版先做状态
判断，在保留有用厚度的前提下，更稳地把产物推向洞察、密度和可执行规则。

## 验证结果

## 边界声明

v0.6.2 不声明跨模型鲁棒性。当前结论来自本仓库 contract tests、pressure records、
OpenCode 旧版 vs 新版 A/B 复测，以及本地 release pack smoke。

v0.6.2 也不完成所有公开包硬化工作。manifest、checksum、完整 validator、从 release
pack 安装的 installer、GitHub Release assets 自动生成，仍保留为后续 work orders。

## 升级影响

如果你只安装和调用 Mindthus skills，本版的主要可见变化是 TVG 输出会更少盲目扩写，
更强调先判断厚度，再提升价值密度。

如果你维护发布流程，本版新增的重点是公开技能包发布边界。以后发布包不能靠人工删文件，
而应该从 allowlist 生成，并用验证证明内部材料没有进入公开包。
