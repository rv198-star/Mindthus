# Mindthus v0.6.3 发布日志

发布日期：2026-06-05

## 版本定位

v0.6.3 是一次 `tplan` 运行时优化小版本。

这版的目标不是给 `tplan` 加更多仪式，而是反过来：保留 Mission、evidence、decision
hook 和 graceful stop 这些真正有价值的能力，同时降低普通运行时的记录密度、脚本调用
和用户认知负担。

一句话：

> tplan 的能力边界不缩水；执行仪式按风险、复杂度、可逆性和恢复需求分层。

## 这次解决了什么

之前的 `tplan` 已经能稳定管理 Mission、任务树、证据和决策钩子，但使用体感偏重：

- 普通动作容易被完整实体化成 Step。
- evidence 和过程记录虽然概念分离，但运行时仍容易显得密。
- decision packet 和 review 容易在低风险选择里产生额外成本。
- 面向用户时，`T1`、`E2` 等内部编号容易让人看不懂当前到底发生了什么。
- 多分支调查时缺少一个安全利用 SubAgent 并行能力的边界。

v0.6.3 把这些问题收束成三条主线：自适应运行、用户可读输出、只读 SubAgent 加速。

## 自适应运行：轻，但不降级

`tplan` 现在把运行成本按风险触发：

- `lite`：低风险、短路径、可逆、目标清楚。只保留恢复所需的最小 Mission 状态。
- `normal`：默认 Mission 工作。有任务树、必要 SubTask/Step、稀疏 evidence 和轻量
  parent alignment。
- `strict`：高风险、长期、多分支、需要审计或人类授权。使用完整 decision packet、
  Mission Review、强 evidence link 和触发式审计。

这三档不是用户需要理解的新开关，而是 `tplan` 内部运行策略。关键规则是：

```text
runtime level may reduce recording density, but it must not weaken key risk triggers.
```

也就是说，lite 只能减少记录密度，不能绕过高影响变更的 alignment、review、
decision hook 或 stop 条件。

## 新增运行支撑

本版新增两个轻量脚本：

- `scripts/init_lite.py`：用一个 active root Task 启动低风险 Mission，不强制先创建 Step。
- `scripts/checkpoint.py`：一次性记录可选本地 log、可选稀疏 evidence，并返回 survey。

Step 也改成延迟实体化：只有当动作需要恢复、验收、回滚、引用 evidence，或者已经膨胀成
多步工作时，才升级成 Step。普通执行痕迹可以留在 task-local log 或 checkpoint 里。

Evidence 继续保持稀疏，只记录：

- acceptance passed / failed
- blocker
- user feedback
- decision
- state transition
- key finding

过程记录不进入 `evidence.jsonl`。

## 用户可读输出

内部编号仍然保留，因为恢复、验证和 evidence link 需要稳定 ID。

但普通用户输出不再默认以 `T1`、`S2`、`E3` 开头，而是先讲人能直接理解的内容：

- 当前目标
- 当前进展
- 已确认事项
- 下一步
- 是否需要用户决定

新增 `scripts/render_user_update.py`，用于生成紧凑中文状态更新。内部编号只在调试、审计、
严格 review、恢复锚点或用户明确要求时作为次要信息出现。

## 只读 SubAgent 加速

本版新增 `resources/subagents.md`，明确：

```text
SubAgents are scouts, not controllers.
```

SubAgent 可以做：

- read files
- search code or docs
- inspect release packages
- compare candidate options
- collect candidate evidence
- perform read-only review

SubAgent 不可以做：

- edit files
- write `mission.json`
- write `evidence.jsonl`
- mutate Task/SubTask/Step nodes
- apply decisions
- make final user-facing conclusions

SubAgent 的输出只是 candidate findings。主 agent 必须验证、合并、判断，然后由主 agent
记录 verified evidence、修改 runtime state、应用 decision 或给出最终用户结论。

这让 `tplan` 能在多分支调查时利用并行发现能力，同时不增加用户心智负担，也不把控制权
交给并行 worker。

## 边界

v0.6.3 不声明跨模型鲁棒性。

v0.6.3 也不声称已经给出干净性能证明。A/B 运行显示新版方向有效，但现场 agent 仍查了
不少 help/source，并有外部连接重试污染，所以这次只把它作为行为验证，而不是严格性能
基准。
