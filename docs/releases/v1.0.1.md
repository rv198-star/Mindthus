# Mindthus v1.0.1 发布日志

发布日期：2026-06-08

## 版本定位

`v1.0.1` 是当前推荐的 1.0 系列安装版本。它包含 `v1.0` 的正式能力，也补上 `v1.0.1` 的使用日志入口。

对用户来说，这个版本做了两件事：

- Mindthus 进入第一版正式发布面：有方法、有授权口径、有忠实度评审入口，也有对“退出是否成立”的检查。
- 每次真实使用都可以留下可脱敏、可校验、可积累的记录，为后续判断哪些方法真的有帮助打底。

核心变化：

> 提供第一版正式可用的判断框架，并让真实使用结果开始沉淀为项目资产。

## v1.0 正式能力

`v1.0.1` 同时包含 `v1.0` 的这些更新：

- 新增 `MPG / 主线-路径博弈`：用于在看见长期主线之后，处理路径波动、对抗力量、载体承受力、暴露预算和熔断点。
- 建立 `Method Fidelity Framework`：约束关键判断动作，不约束判断结论，避免 agent 只做出“看起来像方法”的空壳输出。
- 补齐授权口径：新增 AGPLv3 `LICENSE` 和 `COMMERCIAL-LICENSE.md`，明确开源使用和闭源商业使用的边界。
- 新增忠实度评审自动化入口：生成可复现评审材料，校验评审记录形状、评分字段和证据覆盖。
- 增加退出审查：当输出选择不适用、转交或挑战前提时，评审不能自动放行，必须检查这个退出理由本身是否成立。
- 补跑 SELA 跨模型小样本，关闭单模型发布阻塞；但它仍然不声明所有模型、所有方法都稳定有效。

完整说明见 [Mindthus v1.0 发布日志](https://github.com/rv198-star/Mindthus/blob/v1.0/docs/releases/v1.0.md)。

## v1.0.1 补丁能力

`v1.0` 已经能校验单次 judge 结果，但单次校验不会自动变成项目资产。`v1.0.1` 补上一个最小习惯：

- 记录场景、方法、模型、baseline 分、constrained 分和约束是否有帮助。
- 没有 baseline 时也允许记录 constrained 分，避免真实使用因为门槛太高而不记录。
- 用 `data/fidelity-usage-log.jsonl` 保存 JSONL，方便后续聚合分析。
- 保留边界：脚本只管记录形状，不判断方法价值。

- 新增 `scripts/log-fidelity-usage.py`：追加或校验 Method Fidelity usage log。
- 新增 `data/README.md`：说明默认日志位置和脱敏边界。
- 新增 `docs/internal/fidelity-usage-log.md`：说明 usage log 如何服务 1.0 -> 2.0 的数据飞轮。

## 验证

本补丁应至少通过：

```bash
python3 -m unittest tests.test_v1_0_1_usage_log -v
python3 -m unittest discover -s tests -v
python3 scripts/build-release-pack.py --out /tmp/mindthus-v1.0.1-check --force
```

## 边界

`v1.0.1` 不声明已经具备跨方法的统计结论。它只是让未来的统计结论开始有数据来源。
