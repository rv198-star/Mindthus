# Mindthus v1.1.2 发布日志

发布日期：2026-06-23

## 版本定位

`v1.1.2` 是一次 TPlan runtime control patch release。它把近期围绕长任务回顾、反螺旋和同路径继续授权的设计，收束成一个更清楚的三层控制面：

```text
Snapshot -> Pulse -> Gate
```

这版的重点不是让 agent 每个 active task 都做重度复盘，而是让运行时在真正出现信号时醒来：脚本只观察状态，Pulse 只路由，Gate 才负责判断和 mutation。

## 新增

### TPlan Snapshot / Pulse / Gate

TPlan 公开方法页和 hook 资源现在明确三层职责：

- `Snapshot`：checkpoint、survey、active task、recent evidence、task logs、shared risk 和 validation findings 等只读状态。
- `Pulse`：轻量 Mission-level route note，用 Candidate Collection 和 Gate Arbitration 选择 `next_gate`。
- `Gate`：既有判断中心，包括 `continuation_authorization`、`anti_spiral_audit`、`selection`、`subtraction`、`loopback`、`mission_review`、`risk_assessment` 和 `stop_report`。

短规则是：

```text
Scripts observe. Pulse routes. Gates decide.
```

`mission_pulse.py` 的输出包括 recent evidence summary、active task log summary、evidence-link lint、review trigger candidates、winning candidate、suppressed candidates、arbitration trace 和 shape findings。它是 shape-only 输入，不是健康分、通过/失败判定或 mutation authority。

### 多场景 Mission Pulse 回放

本版固化了多类场景：

- routine checkpoint -> `continue`
- `before_continue` -> `continuation_authorization`
- 同一局部对象第三次触碰 -> `anti_spiral_audit`
- checkpoint batch 没有 fresh acceptance evidence -> `continuation_authorization`
- feedback -> `loopback`
- blocker / failure / interruption / surprise -> `mission_review`
- freeze / handoff / stop -> `mission_review`
- active shared risk -> `health_check`
- branch cleanup / active switch candidate -> `selection`
- Mission `requires_human` -> `stop`
- 无 active task 但要继续 -> `mission_review`

这些测试验证的是路由和优先级仲裁能走通，不把 Gate 内部语义判断交给脚本。

## 修复

### Pulse 仲裁模型

Pulse route 从场景补丁收束为统一候选模型：

```text
Snapshot -> Candidate Collection -> Candidate Shape Validation -> Gate Arbitration -> Pulse Output -> Gate
```

当多个候选同时出现时，Pulse 选择一个 `winning_candidate`，但把其他候选保留在 `suppressed_candidates` 和 `arbitration_trace`。例如 Mission boundary 可以压过 shared risk，但 shared risk 不会被静默丢掉。

### Anti-Spiral runtime gate

`anti_spiral_audit` 现在作为 runtime gate 被 Pulse 唤醒。active task logs 中同一个 `object_id` 出现三次及以上，会生成 `third_touch` 候选；如果日志还显示 `add_layer`、`add_fallback`、`add_rule`、`add_branch` 或 special-case 信号，会同时暴露 `additive_layering`。

这让 TPlan 可以在局部修补开始替代 Mission progress 时更早刹车，而不是等 agent 自己意识到已经进入循环。

### Router wake-up certification gates

`scripts/router-wakeup-ab.py` 增加认证门槛：

- known set 至少 60 个 paired routing moments；
- holdout set 至少 120 个 paired routing moments；
- real-use replay 至少 50 个 paired routing moments；
- overuse stress 必须覆盖 direct、missing-evidence、deterministic、`tvg` 和 `3l5s` 桶。

缺少这些门槛会产生 `minimum-pairs` 或 `minimum-<case-type>` failed check，不能被引用为行为提升认证。

## 边界

- Pulse 不能计算 ROI、分类缺陷、决定证据充分性、重定义验收权威、判断 Mission 是否健康，或自动修改 Mission state。
- `next_gate=continue` 只表示没有机械观察到 review trigger；它不绕过高影响 decision hook、Mission Review 或 stop 边界。
- Anti-Spiral 不禁止明确因果链上的正常调试；如果有 failing test、明确报错和机械验证进展，可以继续沿因果链修复。
- Router certification gates 只提升证据口径诚实度，不声明低频方法 wake-up lift 已被真实使用证明。
