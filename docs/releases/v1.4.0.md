# Mindthus v1.4.0 发布日志

发布日期：2026-06-29

## 版本定位

`v1.4.0` 是一次入口判断能力 release。

它不新增新的主方法，而是把 Mindthus 的核心定位向前推进了一层：agent 不只是要“选对方法”，还要先判断自己是否已经被局部正确、带有倾向性的输入、实现层真相或单一证据信号带偏。

本版正式引入：

- `Frame Fitness Check / 定框适配检查`：共享认知原语。
- `Input Framing Audit / 输入定框审计`：`using-mindthus` 中的方法路由前强约束入口协议。
- primitive activation runtime shape：让脚本可以提醒结构化输出形状，但不替 agent 判断语义正确性。

核心目标：

```text
在进入判断之前，先检查当前问题是否已经被提问方式绑到错误层级。
```

## 安装入口

普通用户继续按 release asset 安装；plugin mode 下载 plugins 包，skills-pack mode 下载 skills 包：

```bash
VERSION=1.4.0
curl -L \
  -o /tmp/mindthus-plugins-${VERSION}.tar.gz \
  "https://github.com/rv198-star/Mindthus/releases/download/v${VERSION}/mindthus-plugins-${VERSION}.tar.gz"
rm -rf /tmp/mindthus-plugins
mkdir -p /tmp/mindthus-plugins
tar -xzf /tmp/mindthus-plugins-${VERSION}.tar.gz -C /tmp/mindthus-plugins --strip-components=1

curl -L \
  -o /tmp/mindthus-skills-${VERSION}.tar.gz \
  "https://github.com/rv198-star/Mindthus/releases/download/v${VERSION}/mindthus-skills-${VERSION}.tar.gz"
rm -rf /tmp/mindthus-skills
mkdir -p /tmp/mindthus-skills
tar -xzf /tmp/mindthus-skills-${VERSION}.tar.gz -C /tmp/mindthus-skills --strip-components=1
```

本次 release asset 名称是 `mindthus-plugins-1.4.0.tar.gz` 和 `mindthus-skills-1.4.0.tar.gz`。

## 这次解决了什么

### 局部正确接管全局判断

本版针对一种比简单错误更累人的失败模式：agent 沿着一个局部正确的框架持续自洽，最后把全局判断带偏。

例如：

```text
Skills 本质上就是提示词。
```

这句话在实现层可能局部成立，因为许多 skill 确实会以文本形式进入上下文。但如果 agent 直接接受这个 framing，就会把实现载体误当定义，把注意力管理误当架构价值，把局部机制误当整体解释。

`Frame Fitness Check` 的作用，就是在回答前先判断这个 frame 是否适配当前问题层级。

### Input Framing Audit

`using-mindthus` 现在有一个更明确的入口协议：出现 framing-risk 时，先做输入定框审计，再路由到具体方法。

审计至少包含这些内部结果：

- `true_question`
- `packed_premises`
- `layer_risks`
- `frame_status`
- `reframed_question`
- `routing_decision`

`frame_status` 的取值为：

- `clean`
- `biased`
- `overloaded`
- `malformed`

这些字段不是为了让 agent 对用户展示模板，而是为了让下游动作真的改变：正常路由、先指出偏置、先拆命题、先纠正问题，或先取证。

### Framing-risk signals, not keyword rules

本版明确：这不是关键词触发器。

`本质上`、`其实就是`、`无非是`、`正因为我是...` 这类词只是高置信线索。没有这些词，只要出现打包结论、层级偷换、局部机制冒充整体解释，也应触发。

反过来，有这些词也不代表一定要审计。低风险范围说明、审美偏好、产品目标、风险姿态或权限边界，通常应保留为用户目标函数，而不是当成偏见抹掉。

## 边界

- 本版不新增 `frame` 或 `anti-sycophancy` 独立 skill。
- 本版不让 agent 对所有任务先跑一套输入审计模板。
- No frame-risk signal, no frame check.
- No execution impact, omit the frame check.
- 输入审计不能替代证据获取、领域事实、运行时验证或正式方法分析。
- 用户价值、偏好、审美和风险姿态是判断约束，不能被当成偏见抹掉。
