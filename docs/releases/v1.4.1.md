# Mindthus v1.4.1 发布日志

发布日期：2026-06-29

## 版本定位

`v1.4.1` 是一次输入定框审计强化 release。

它延续 v1.4.0 的方向，但把真实案例里更有效的提示词纪律沉淀回 `using-mindthus`：agent 的第一任务不是顺着用户的话回答，而是先判断当前问题是否已经被提问方式带到错误层级。

本版重点补强三件事：

- `Explanatory Authority Check / 解释权校准`：局部正确不自动拥有整体解释权。
- `Dominant Carrier Check / 主导承载校准`：先看真正承载稳定性、价值或决策权的机制是什么。
- `System Subject Check / 系统主体校准`：系统问题不能只按模型局部注意力或单一实现载体解释。

## 安装入口

普通用户继续按 release asset 安装；plugin mode 下载 plugins 包，skills-pack mode 下载 skills 包：

```bash
VERSION=1.4.1
curl -L \
  -o /tmp/mindthus-plugins-${VERSION}.tar.gz \
  "https://github.com/rv198-star/Mindthus/releases/download/v${VERSION}/mindthus-plugins-${VERSION}.tar.gz"
rm -rf /tmp/mindthus-plugins
mkdir -p /tmp/mindthus-plugins
tar -xzf /tmp/mindthus-plugins-${VERSION}.tar.gz -C /tmp/mindthus-plugins --strip-components=1

curl -L \
  -o /tmp/mindthus-skills-${VERSION}.tar.gz \
  "https://github.com/rv198-star/Mindthus/releases/download/v${VERSION}/mindthus-skills-${VERSION}.tar.gz"
rm -rf /tmp/mindthus-skills
mkdir -p /tmp/mindthus-skills
tar -xzf /tmp/mindthus-skills-${VERSION}.tar.gz -C /tmp/mindthus-skills --strip-components=1
```

本次 release asset 名称是 `mindthus-plugins-1.4.1.tar.gz` 和 `mindthus-skills-1.4.1.tar.gz`。

## 这次解决了什么

### 把有效提示词变成通用入口纪律

此前案例里真正生效的关键不是“多加几个怀疑句”，而是改变回答顺序：

```text
先审题，后答题。
先拆前提，后接推理。
先判层级，后判对错。
```

v1.4.1 把这套顺序写入 `Input Framing Audit`：当出现 framing-risk 时，先识别真实问题、隐含前提、局部成立性、层级偷换和重述问题，再正式回答。

### 防止局部正确获得整体解释权

本版强化了一个通用判断：一个说法在实现层成立，不代表它能定义系统本质；一个机制能解释局部现象，也不代表它拥有全局解释权。

这用于避免 agent 在面对专业口吻、权威身份、实现细节或漂亮的还原论时，过早承认对方的叙事框架，然后只在框架内部做温和修正。

### 识别系统主体，而不是只看显眼载体

系统类问题里，最显眼的载体不一定是主导机制。文本、模型、脚本、workflow、state、evidence gate、authority boundary、repeatability 和 value carrier 可能分别承担不同作用。

v1.4.1 要求 agent 先判断：当前对象真正由什么承载稳定性、确定性、复用性和解释权。这样可以减少把“局部实现介质”误当“系统主体”的输出。

### 可确认本地热更新是否生效

新增 `scripts/log-mindthus-runtime.py`：

- 检查 repo、本地 marketplace、Codex cache 三处 runtime 文件。
- 输出关键文件的 sha256、bytes、mtime。
- 检查核心 marker 是否都存在。
- `--strict` 下发现 marker 缺失或 hash 不一致会返回非零退出码。

这个脚本解决的是“我以为更新了，但实际跑的可能还是旧版本”的确认问题。

## 实测校准样例

本版的核心测试来自一个真实输入：

```text
我是做 Agent 开发，当然更加明白 skills 就是提示词，所谓的脚本门禁也是提示词，
本质都是文本注入提高 LLM 短时间注意力范围。你如何评价这句话？
```

在 `019f1753` 一类回放里，模型第一轮已经能指出“局部正确不等于整体本质”，但在用户继续追问
“我一直在说 SKILLS，和 Agent 啥关系”之后，仍可能过度回撤，最后承认“如果限定在 skills，
skills 基本就是 prompt injection”。这说明当前版本仍未完全达到预期：它已经能更早发现局部正确，
但还不总能在 scope correction 之后守住定义权。

目标 95 分参考回答不是简单唱反调，而是更准确地区分两种用法：

> 对 SKILLS 来说，prompt/context 注入是一种真实且常见的轻量用法，但它不能自动获得本质解释权。
> 关键要看谁控制目标结果：一种是 LLM 主导，脚本服务模型，skill 主要负责把上下文安排给模型；
> 另一种是脚本主导，LLM 服务脚本，skill 把流程、校验、复用和 AI 介入时机封装成任务能力。
> 前者更常见，后者在稳定完成任务、可复用、可校验时往往价值更高。好的定义不能抹掉任一侧：
> SKILLS 是可触发的任务能力封装，文本注入是常见接口；当脚本/规程承担结果控制权时，
> 它们才是稳定性和确定性的主承载。如果把 SKILLS 定义成 prompt，会把优化方向从稳定完成任务
> 带偏到更聪明的话术。

这段参考回答只作为 `using-mindthus` 的校准样例，不作为针对 SKILLS 场景的硬编码结论。
通用规则仍然是：先还原整体对象，再判断哪个局部机制是否拥有定义权。

## 边界

- 本版不新增独立 skill，也不把输入审计变成所有任务的模板。
- 本版不针对某个 skills/prompt 示例写死结论；新增规则是通用的定框、解释权和系统主体校准。
- 本版不声明已稳定达到 95+ 行为水平；当前版本仍未完全达到预期，尤其在用户连续纠正范围时仍需继续观察。
- runtime 日志脚本只验证文件版本一致性，不验证模型行为质量。
- 输入审计仍不能替代证据获取、领域事实、运行时验证或正式方法分析。
