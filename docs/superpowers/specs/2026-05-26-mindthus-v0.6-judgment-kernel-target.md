# Mindthus v0.6 Judgment Kernel Target

## Status

This is a direction record, not an implementation plan.

It captures the current target for Mindthus v0.6: improve the judgment philosophy /
method kernel from a useful method set toward a more coherent, elegant, and systematic
judgment framework.

The target has been narrowed from "add a full meta-framework" to a smaller kernel:

> Find the right judgment point.
>
> Deepen the artifact where value is thin.
>
> Stop when Mindthus is not needed.

Do not treat the candidate solution ideas below as approved architecture. They describe
the defect surface around this kernel, not the final design.

## Target

Mindthus should not become a cognitive memory product, a retrieval system, or a full
harness runtime in v0.6.

The v0.6 target is narrower:

> Raise the judgment philosophy / method kernel from roughly `60%-70%` maturity toward
> `90%+`.

This means Mindthus should become more reliable as a reusable thinking foundation:

- clearer about what kind of judgment problem is active;
- clearer about when a method should or should not be used;
- clearer about how methods relate when their recommendations conflict;
- clearer about its own failure modes;
- easier for a smart non-specialist to understand without internal jargon.

## Current Strengths

Mindthus already has a useful core:

- `3L5S` handles problem discovery, definition, and landing.
- `SELA` handles long-term system-efficiency versus local-advantage judgment.
- `EDSP` handles structural ambiguity, false binaries, and scenario placement.
- `WAE` handles control boundaries between workflow, agentic judgment, and evidence.
- `TVG` handles bounded value deepening for thin but polished artifacts.
- `tplan` handles mission-oriented long-task runtime discipline.
- Anti-Spiral prevents repeated local repair from replacing objective progress.

These methods are individually useful and increasingly well-bounded.

## v0.6 Architecture Hygiene Goal

Mindthus should treat repeated cross-cutting mechanisms as shared primitives rather than
copying them into each skill.

This is similar to an AOP concern in software design: logging, authorization, or
transactions should not be hand-written inside every business function. In Mindthus,
cross-cutting reasoning controls should not be reinvented separately inside `SELA`,
`EDSP`, `WAE`, `TVG`, `3L5S`, or `tplan`.

The goal is not to add a new meta-framework. The goal is to prevent architecture decay:

- preserve each skill's mainline;
- keep shared pressure / brake / evidence mechanisms consistent;
- avoid duplicate vocabulary for the same reasoning concern;
- prevent guardrails from quietly becoming new judgment centers;
- make future maintenance orderly.

Early shared primitive candidates:

- `Minimal Sufficient Lens`: prevent over-methodizing simple work.
- `Evidence / Claim Ceiling`: prevent unevidenced claims from becoming confident
  conclusions.
- `Perspective Pressure`: prevent single-perspective self-consistency.
  - `Multi-Role Pressure` is one implementation form.
  - `Adversarial Incentive Check` is one high-value trigger when real stakeholder
    incentives may distort judgment.
- `Anti-Spiral`: prevent local repair loops from replacing objective progress.
- `No Abstract Jargon Wall`: prevent internal method vocabulary from replacing user
  understanding.

These are v0.6 design candidates, not approved implementation surfaces. A shared
primitive should be extracted only when it is repeated across methods, protects the
mainline, and can stay smaller than a standalone skill.

## v0.6 Kernel Shape

### 1. Find The Right Judgment Point

Mindthus complements LLMs when ordinary generation is no longer enough and the model
needs stable analysis, judgment, or decision support.

The first job is not to route every user request into a method. Many requests are ordinary
LLM tasks and should be answered directly. Mindthus should intervene when the active
failure risk is a misplaced judgment point:

- the model is solving before the real problem is known;
- the model is treating a false binary as a real binary;
- the model is converting a long-term trend into an immediate action too early;
- the model is letting workflow, agentic judgment, or evidence control the wrong part;
- the model is continuing a local repair loop instead of returning to the objective.

This is the shared role of `3L5S`, `SELA`, `EDSP`, `WAE`, `tplan`, and Anti-Spiral.

### 2. Deepen The Artifact Where Value Is Thin

Once the right bounded artifact exists, Mindthus should help the LLM make it materially
more useful.

The failure here is not wrong routing, but thin output:

- structure looks complete, but judgment is shallow;
- language is fluent, but evidence and trade-offs are weak;
- the artifact reads well, but downstream users still have to invent the missing logic;
- the output is longer, but not more decision-useful.

This is the central role of `TVG`: deepen only where practical value can increase.

### 3. Stop When Mindthus Is Not Needed

Mindthus should not become a universal task gateway.

It should stay out when:

- the user task is clear, low-risk, and directly executable;
- the missing input is facts, files, runtime proof, domain research, or current data;
- ordinary LLM capability is enough;
- adding a method would increase ceremony without improving the decision;
- the work is outside authority and Mindthus can only help frame questions, not decide.

This stop rule is not a third production function. It is the brake that keeps the first
two functions useful.

## Current Defects Under This Kernel

### 1. The "Find The Judgment Point" Function Is Still Too Distributed

Mindthus has strong principles, but they are still distributed across skills and docs.

Examples:

- judgment before execution;
- target function before method selection;
- real problem before solution;
- control boundary before automation;
- evidence before confidence;
- method as cost reducer, not ceremony;
- downstream action value over structural completeness.

These are present as repeated instincts, but not yet expressed as a compact top-level
entry discipline.

The previous idea of a "Top-Level Judgment Constitution" is useful as a defect label,
but the better v0.6 shape may be lighter: a short entry discipline that tells the model
when it has reached a hard judgment point.

### 2. The "Judgment Object" Is Not Yet Clear Enough

Mindthus has skill routing, but it does not yet fully classify the object being judged.

Before asking "which skill should I use?", Mindthus should help identify whether the
active object is:

- a problem-definition failure;
- a goal-function conflict;
- a false binary;
- a control-boundary mismatch;
- a thin artifact;
- a local-repair spiral;
- a continue / stop / switch decision;
- an evidence / confidence mismatch.

Without this map, method selection can still feel like choosing among named tools rather
than diagnosing the judgment object.

This should not become a large taxonomy. The desired shape is a minimal set of
high-signal judgment-point types.

### 3. The "Deepen Thin Value" Function Is Strong But Too Isolated

`TVG` is already the clearest answer to thin artifacts, but its relationship to the rest
of Mindthus can be sharper.

Open issues:

- when should a thin artifact go to `TVG` versus back to `3L5S`, `EDSP`, or `WAE`?
- when is the artifact thin because content is weak, and when is it thin because the
  judgment point was wrong earlier?
- when should `TVG` block rather than deepen because evidence, domain input, or runtime
  proof is missing?

The v0.6 kernel should make `TVG` visibly the second main function of Mindthus, not just
one skill among many.

### 4. The Stop Rule Is Not Yet First-Class

Mindthus already has "minimal sufficient lens", low-risk boundaries, TVG block rules,
and Anti-Spiral, but the stop rule is still scattered.

Known stop cases:

- direct execution is enough;
- facts should be gathered before reasoning;
- the task is outside authority;
- method use would add ceremony;
- the same local object is being repaired repeatedly;
- a clean structure is making the output thinner.

The v0.6 design should make "do not use Mindthus here" as legitimate as selecting a
skill.

### 5. Cross-Method Arbitration Remains Unsettled

Each method has improving local boundaries, but Mindthus does not yet have a mature rule
for conflicts between methods.

Examples:

- `SELA` may see the long-term system path, while `WAE` may restrict immediate control
  because the work is irreversible or high-blast-radius.
- `TVG` may propose another value-deepening pass, while Anti-Spiral may say the work is
  becoming a local repair loop.
- `EDSP` may produce a structural direction, while evidence boundaries prevent turning
  that direction into a confident claim.
- `3L5S` may ask for more problem definition, while the user has already supplied a
  sufficiently clear target and wants execution.

Mindthus needs a more elegant way to decide when a method should dominate, defer, degrade,
or stop.

### 6. System-Level Failure Modes Are Not Yet Unified

Individual skills already describe common misuse, but Mindthus does not yet expose a
single failure map for the whole method system.

Known failure modes include:

- premature execution;
- method overuse;
- clean-but-thin output;
- evidence theater;
- local repair spiral;
- false binary;
- efficiency dogma;
- control mismatch;
- template completion replacing judgment;
- internal jargon replacing user understanding.

This map should help both users and agents recognize when Mindthus itself is being used
poorly.

### 7. Threshold Cases Are Still Sparse

Mindthus has examples and A/B pressure tests, but not yet a systematic threshold casebook.

The most valuable cases are not obvious wins. They are boundary cases where:

- the right method is easy to misuse;
- two methods appear plausible;
- the answer is not changed, but the boundary becomes clearer;
- the method must stop, degrade, or hand off instead of continuing.

Threshold cases are needed to make the judgment kernel feel real, not merely well-worded.

## Design Constraint

The solution should not be a large patch pile.

The previous candidate list of "constitution / object map / arbitration / failure map /
casebook" is useful as a defect map, but too additive as an architecture.

The v0.6 design should search for a more elegant system shape:

- fewer primitives rather than more labels;
- shared primitives for repeated cross-cutting controls, not scattered guardrails copied
  into several skills;
- one coherent route from "is this a hard judgment point?" to "which minimal intervention
  is useful?";
- no broad expansion into memory, retrieval, or runtime harness infrastructure;
- no heavy mandatory flow for simple tasks;
- no requirement that `3L5S` become the universal entry point;
- no internal jargon wall.

## Non-Goals

v0.6 should not:

- turn Mindthus into a cognitive memory product;
- build a retrieval or context database layer;
- build a full harness runtime;
- add many new skills before the existing method kernel is made more coherent;
- make every task run through a fixed master workflow;
- claim that philosophical completeness replaces evidence, domain research, runtime proof,
  or user judgment.

## Open Question

What is the smallest elegant structure that can unify:

- finding the right judgment point;
- deepening thin bounded artifacts;
- stopping when ordinary LLM execution is enough;
- method selection;
- cross-method conflict handling;
- and failure-mode awareness,

without turning Mindthus into a large meta-framework?
